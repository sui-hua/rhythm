# 计划时间全层级写入 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 新建和后续补齐计划时，让 `plans`、`monthly_plans`、`daily_plans` 三层记录都写入明确的 `task_time` 和 `duration`。

**Architecture:** 数据库三张表已经存在 `task_time` 和 `duration` 字段，本计划只调整 Direction 前端写入链路，不新增 schema。顶级计划创建时把表单时间写入顶级计划和同时创建的月计划；批量创建/更新日计划时从当前月计划读取默认时间，月计划缺值时回退到顶级计划时间；用户单独修改过的月计划或日计划时间保持原值。

**Tech Stack:** Vue 3 Composition API, Vitest, Supabase service wrapper, Direction composables

---

## 文件结构

- 修改: `src/views/direction/composables/useDirectionGoals.js`，统一顶级计划创建、范围扩展、顶级时间编辑时的月计划时间写入规则。
- 修改: `src/views/direction/composables/useDirectionBatch.js`，让批量创建和批量更新日计划使用月计划默认时间，月计划缺值时回退到顶级计划时间。
- 修改: `src/views/direction/composables/__tests__/useDirectionGoals.spec.js`，覆盖新建顶级计划时月计划同步写入时间，以及范围扩展创建月计划时继承时间。
- 修改: `src/views/direction/composables/__tests__/useDirectionBatch.spec.js`，覆盖新建日计划和更新已有日计划时写入默认时间。
- 查看: `database/current-structure.md`，确认 `plans`、`monthly_plans`、`daily_plans` 已包含 `task_time`、`duration` 字段。

### Task 2: 实现月计划继承顶级计划时间

**Files:**
- Modify: `src/views/direction/composables/useDirectionGoals.js`

- [ ] **Step 1: 在 `isSubmitting` 后新增时间解析 helper**

在 `const isSubmitting = ref(false)` 后新增：

```javascript
  const resolvePlanTiming = (source = {}, fallback = {}) => ({
    task_time: source.task_time ?? fallback.task_time ?? '09:00',
    duration: source.duration ?? fallback.duration ?? 30
  })
```

- [ ] **Step 2: 修改 `updateGoalData()` 创建缺失月计划的 payload**

在 `const currentTitle = ...` 后新增：

```javascript
      const currentTiming = resolvePlanTiming(
        { task_time: newTaskTime, duration: newDuration },
        goalToUpdate
      )
```

将 `db.monthlyPlans.create({ ... })` 内的：

```javascript
            task_time: null,
            duration: null
```

替换为：

```javascript

            duration: currentTiming.duration
```

- [ ] **Step 3: 修改 `updateGoalData()` 对已有月计划的补齐规则**

将当前只更新标题的逻辑：

```javascript
      const updatePromises = []
      if (newTitle !== undefined) {
        updatePromises.push(...existingMonthlyPlans.map(mp => db.monthlyPlans.update(mp.id, { title: newTitle })))
      }
```

替换为：

```javascript
      const updatePromises = existingMonthlyPlans
        .map(mp => {
          const payload = {}
          if (newTitle !== undefined) payload.title = newTitle
          if (newTaskTime !== undefined && !mp.task_time) payload.task_time = currentTiming.task_time
          if (newDuration !== undefined && !mp.duration) payload.duration = currentTiming.duration
          return Object.keys(payload).length > 0 ? db.monthlyPlans.update(mp.id, payload) : null
        })
        .filter(Boolean)
```

这条规则只补齐空值，不覆盖已经单独编辑过时间的月计划。

- [ ] **Step 4: 修改 `handleAddGoal()` 创建月计划的 payload**

将 `monthlyPlanData` 内的：

```javascript
          task_time: null,
          duration: null
```

替换为：

```javascript
          task_time: planData.task_time,
          duration: planData.duration
```

## 影响范围

- 不修改 Supabase schema，因为 `database/current-structure.md` 已记录三层计划表都有 `task_time` 和 `duration` 字段。
- 不引入全局继承服务，只在 Direction 当前写入口补齐 payload，保持改动面集中。
- 不批量覆盖已有非空月计划和日计划时间，避免破坏用户已经单独调整过的时间。
- 历史数据已完成线上处理，本计划不要求执行者再次修改现有数据。
