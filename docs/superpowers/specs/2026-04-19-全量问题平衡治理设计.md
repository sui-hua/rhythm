# Rhythm 全量问题平衡治理设计

## 背景

基于 `2026-04-19-项目审查报告`，当前项目的问题不是单一层面的缺陷，而是多条问题链同时跨越产品、交互、性能与工程边界：

1. `所向 -> 时序` 的目标闭环没有真正落到每日执行层。
2. `task / daily_plan / habit` 在时序页同时存在，但语义和交互边界不清。
3. 移动端任务新增、编辑、反馈存在明显阻塞。
4. `Direction` 的请求、缓存、状态边界不稳定，已经反映到失败单测。
5. 数据访问层同时存在 `db` 与 `safeDb` 两套入口，且错误契约不一致。

如果按纯前端体验或纯底层重构单线推进，都容易让另一侧的问题持续吞噬收益。因此本设计采用“平衡推进 + 垂直切片”的方式，按完整业务链分批修复。

## 目标

1. 用三批改造完成当前高优先级问题治理，而不是只做局部止血。
2. 第一批同时解决最影响用户体验和最影响稳定性的交叉问题。
3. 保留现有对外产品语言：`时序 / 习惯 / 所向 / 总结`。
4. 把 `safeDb` 合并进唯一 `db`，形成单一正式数据访问入口。
5. 在不大改一级命名和整体产品叙事的前提下，把时序页做成真正的今日执行中心。

## 非目标

1. 第一批不修改一级导航和对外命名，不把 `时序 / 习惯 / 所向` 改成新话术。
2. 第一批不上收件箱 / 待安排层。
3. 第一批不重做整体视觉系统，不做大规模风格统一。
4. 第一批不改 Summary 闭环设计，只保证它不继续拖后腿。
5. 本设计不覆盖后端新服务形态的详细接口协议，事务化写入放在第三批增强阶段。

## 总体策略

采用 **平衡推进 + 垂直切片**。

含义是：

1. 每一批都必须同时包含用户可感知的变化和底层稳定性改造。
2. 每一批都必须能独立上线。
3. 不按“先全部前端、再全部架构”推进，而按完整问题链切片推进。

对比其他路径：

- 不采用“体验先行型”，因为 `Direction` 请求与缓存边界会继续制造错误和返工。
- 不采用“底层收口型”，因为短期用户感知过弱，不能快速建立收益。
- 不采用“大一统重构型”，因为单批风险过高，且会让范围失控。

## 第一批、第二批、第三批的边界

### 第一批：结构性止血批

目标：先把最伤用户、也最伤稳定性的主链问题压住，同时处理已经确认“移动端根本不可用”的结构性页面问题。

范围：

1. 合并 `safeDb` 到唯一 `db`
2. 重构 `Direction` 请求 / 缓存 / reset
3. 落地时序页双层结构最小版
4. 修复时序页移动端任务新增与编辑路径
5. `Habits` 拆成 `DesktopPage / MobilePage`
6. `Direction` 拆成 `DesktopPage / MobilePage`
7. 统一高频写操作反馈
8. 修正 Year 页数据来源错误，保证年视图结果正确

明确不做：

1. 不改一级命名
2. 不改信息架构文案
3. 不上收件箱
4. 不做 Summary 闭环增强
5. 不做视觉系统统一大改

### 第二批：收口批

目标：把“结构勉强成立但解释不清、维护不稳”的部分拉直。

范围：

1. 信息架构文案与页面说明统一
2. onboarding 与空状态补齐
3. 通知权限触发时机后移
4. 习惯页性能收口
5. `db` 契约统一后的页面级单例状态回收

与第一批的关系：

1. 第一批先完成 `Habits / Direction` 的分端页面壳拆分，解决“主内容不可达”的结构性问题。
2. 第二批的“习惯页性能收口”是在移动端和桌面端页面壳边界已经拉开的前提下，对共享数据层和壳内交互再做局部 patch、日志分页、watch 收敛，不与第一批重复。
3. 第二批的“页面级单例状态回收”依赖第一批已经完成 `db` 单入口合并和关键页面主链稳定，否则会在双入口和旧缓存模式上继续返工。

### 第三批：增强批

目标：把项目从“修顺”推进到“成熟”。

范围：

1. 收件箱 / 待安排层
2. Summary 真正闭环化
3. 视觉语法统一
4. Direction 多表写入事务化
5. 服务层用户隔离双保险

## 核心约束

### 1. 对外业务语义不变

项目对外仍使用：

- `时序`
- `习惯`
- `所向`
- `总结`

本次改造不会引入新的一级产品话术去替换它们。

### 2. 时序页保留三类内容

时序页仍同时展示：

- `task`
- `daily_plan`
- `habit`

它不是任务页，而是“今日执行中心”。

### 3. 时序页左侧只做弱分组

时序页左侧不使用强命名分组标题，例如不直接把区域标题改成“推进项 / 临时项 / 维持项”。

采用弱分组方式：

1. 用间距、细分隔组织列表
2. 用来源小标签标识条目类型
3. 保留现有产品语境，不把用户带进新的一套术语系统

内部设计上，我们仍把三类内容区分为不同来源，但这套语义只作为实现和设计依据，不作为对外命名替换。

来源小标签在第一批中的推荐写法：

- `所向`
- `时序`
- `习惯`

不引入新的对外标签命名。

## 模块职责重构

### 时序页（Day）

时序页的职责是“今日执行中心”，不是单纯混排列表。

它需要同时承载：

1. 来自 `daily_plan` 的所向相关执行项
2. 来自 `task` 的普通时序任务
3. 来自 `habit` 的习惯项

#### 呈现结构

采用双层方案：

1. 左侧栏：弱分组列表
2. 右侧：时间轴混排

#### 左侧栏

左侧栏职责不是再复制一份时间轴，而是承担“今日执行索引层”。

要求：

1. 条目按来源自然聚类，但不使用强标题分组
2. 每个条目显示：
   - 标题
   - 时间
   - 来源小标签
   - 完成状态
3. `daily_plan` 必须展示所属目标信息
4. `habit` 必须能被一眼识别，不伪装成普通任务

#### 右侧时间轴

右侧时间轴继续保留现在的核心价值：按时间看今天会发生什么。

要求：

1. 三类内容都进入统一时间轴
2. 每张卡片增加来源标识
3. `daily_plan` 卡片显示所属目标
4. `habit` 卡片显示习惯身份
5. 未安排时间的项进入单独“未安排”区域，而不是混入时间主干

#### 左右映射

两边必须是同一份今日执行数据的两种视图。

要求：

1. 点左侧条目，右侧滚动并高亮对应卡片
2. 点右侧卡片，左侧定位对应条目
3. 不允许两边使用不同的分组口径或不同的数据排序规则

### 所向模块职责

所向页仍然是目标编排中心。

职责：

1. 管长期目标
2. 管月度规划范围
3. 管与目标相关的日计划承诺项
4. 提供“这些承诺项今天在哪里执行”的视角

它不再承担重复请求、重复 watcher 和缓存生命周期不清的状态组织方式。

### 总结页（Summary）

第一批不重构其产品定位，只要求：

1. 不继续作为错误契约混乱的例外
2. 写操作反馈与其它高频模块一致

Summary 闭环增强放入第三批。

### 习惯页（Habits）

当前已确认移动端“只能显示侧边栏”，主内容不可达，因此它不是单纯体验优化问题，而是结构性不可用。

可选方案对比：

- 方案 A：同一页面下拆 `DesktopLayout / MobileLayout`
  - 优点：改动更小，保留单页面入口
  - 缺点：容易继续在同一页面壳里积累 `isMobile` 分支，后期走移动端优先时会再次受限
- 方案 B：同一路由下拆成不同页面壳
  - 优点：桌面和移动边界最清晰，更适合后期走移动端优先
  - 缺点：第一批改动更大，需要强约束共享逻辑不复制

第一批采用 **方案 B：同一路由下拆成不同页面壳**。

选择 B 的原因：

1. 当前移动端问题已经不是简单布局适配，而是结构性不可用。
2. 已明确存在后续“移动端优先”的演进方向，提前拉开页面壳边界更合适。
3. 如果继续采用方案 A，短期能修，但后续大概率还会再次拆壳。

要求：

1. 路由仍保持 `/habits`
2. 页面入口根据端形态分发到：
   - `HabitsDesktopPage`
   - `HabitsMobilePage`
3. 数据查询、写操作、统计逻辑保持共享
4. 不允许桌面布局继续通过 `isMobile` 强行压缩到手机上

移动端目标不是一步做到最佳体验，而是先满足：

1. 主内容可达
2. 可切换习惯对象
3. 可新增 / 编辑习惯
4. 删除最后一个习惯后有明确空状态

### 所向页（Direction）

当前已确认移动端“只能显示侧边栏”，主内容不可达，因此同样属于第一批必须处理的结构性阻塞项。

可选方案对比：

- 方案 A：同一页面下拆 `DesktopLayout / MobileLayout`
  - 优点：接入快，入口结构变化小
  - 缺点：桌面三栏工作台和移动端结构长期共存于同一页面壳，复杂度会持续上升
- 方案 B：同一路由下拆成不同页面壳
  - 优点：最适合 `Direction` 这类桌面和移动结构差异很大的页面
  - 缺点：第一批范围会扩大，需要明确共享层边界

第一批同样采用 **方案 B：同一路由下拆成不同页面壳**。

选择 B 的原因：

1. `Direction` 的桌面端本质上是工作台式三栏结构，移动端不会长期复用同一壳层。
2. 当前移动端已不可用，继续保留同壳混合只会推迟结构性调整。
3. 方案 B 更符合后续移动端优先演进。

要求：

1. 路由仍保持 `/direction`
2. 页面入口根据端形态分发到：
   - `DirectionDesktopPage`
   - `DirectionMobilePage`
3. 共享 `db`、共享目标数据模型、共享写操作反馈规范
4. 不允许继续在同一布局里通过 `isMobile` 混合桌面三栏结构与移动端结构

移动端第一批只要求达到“可用”：

1. 能进入目标主内容
2. 能切换目标
3. 能查看月度编排主内容
4. 能完成新增 / 编辑核心操作

不要求第一批就把移动端所向页做到最终形态最优。

## 统一数据访问层设计

### 为什么只保留一个 `db`

当前问题不在于文件名，而在于职责混乱：

1. 一部分页面直接依赖 `db`
2. 一部分页面依赖 `safeDb`
3. `safeDb` 有时返回 `null`，有时抛异常
4. 页面层无法可靠区分“无数据”和“请求失败”

因此最终只保留一个正式数据访问层：`db`。

### 最终形态

保留 `src/services/database.js` 作为唯一出口。

统一 `db` 的职责：

1. 提供查询与写入
2. 统一错误契约
3. 统一用户作用域过滤策略
4. 做必要的数据结构适配

明确不承担：

1. 不直接弹 toast
2. 不吞错误
3. 不替页面决定交互反馈

### 错误契约

第一批统一为：

- 成功：直接返回数据
- 失败：统一 `throw`

页面 action / composable 决定：

1. 是否显示 toast
2. 使用什么文案
3. 是否做局部回滚
4. 是否继续刷新界面

### 用户作用域

第一批先把“只有一个 `db`”和“错误契约一致”落下。

如果当前底层已依赖 RLS，第一批不要求一次性把所有查询改成显式 `user_id` 过滤，但需要为后续增强留出统一入口，不再允许页面绕过 `db` 直接自己查。

第三批再补“代码层过滤 + RLS”双保险。

## 时序页统一 view model

为了让左侧弱分组和右侧时间轴消费同一份数据，第一批引入统一的执行项 view model。

推荐字段：

```ts
type DayExecutionItem = {
  id: string
  type: 'task' | 'daily_plan' | 'habit'
  title: string
  time: string
  startHour?: number
  duration?: string
  completed: boolean
  sourceLabel: string
  sourceMeta?: {
    planId?: string
    planTitle?: string
  }
  original: unknown
}
```

`type` 与 `sourceLabel` 的映射在第一批中必须显式固定：

- `task -> 时序`
- `daily_plan -> 所向`
- `habit -> 习惯`

不允许在不同页面或组件里各自推断这层关系。

约束：

1. 左侧和右侧都只消费这份标准化数据
2. 来源标签来自现有产品语言，不引入新命名体系
3. `daily_plan` 的 `sourceMeta.planTitle` 为第一批必需字段

## 第一批详细设计

### 第一批实施顺序

第一批内部不是 8 个并列任务，推荐顺序如下：

1. `db` 合并
2. `Direction` 请求 / 缓存 / reset 重构
3. `Habits / Direction` 分端页面壳拆分
4. 时序页统一 view model 与双层结构最小版
5. 时序页移动端任务流收口
6. 高频写操作反馈统一
7. Year 页数据修正

依赖关系说明：

1. 时序页双层结构不强依赖 `db` 合并完成后才能开始界面开发，但它依赖统一的执行项模型；而统一模型最终要落在单一 `db` 返回契约之上，所以建议 `db` 合并先做。
2. `Direction` 分端页面壳拆分依赖 `Direction` 请求 / 缓存主链先稳定，否则只是把不稳定状态复制到两个页面壳。
3. 时序页移动端任务流收口依赖双层结构的条目模型和入口关系先确定。
4. 高频写操作反馈放在后面统一收口，是为了避免前面几项在旧交互流上重复接 feedback。
5. Year 页数据修正可并行，但从风险控制看更适合放在第一批尾部完成。

### 1. `db` 合并

第一批要做的不是“保留两套实现再慢慢迁”，而是完成单入口收口。

做法：

1. 把 `safeDb` 中仍有价值的包装逻辑吸收到 `db`
2. 删除“失败返回 null”的模式
3. 页面统一改为只从 `db` 取数据
4. 不再新增任何对 `safeDb` 的引用

验收标准：

1. 代码搜索中业务代码不再依赖 `safeDb`
2. 查询失败不会再被伪装成空数组或空对象

### 2. `Direction` 请求 / 缓存 / reset

第一批重点收口这条链：

1. 默认月份选择和 `dailyPlans` 预加载只保留一个入口
2. watcher 不重复承担同一职责
3. 同一 `monthlyPlanId` 的请求可复用，不重复发起
4. 增加 `resetDirectionState()`
5. 在登出 / 用户切换时重置缓存

验收标准：

1. 目标切换不再重复请求同一 `dailyPlans`
2. 旧请求不会覆盖新状态
3. 当前 `useDirectionFetch.spec.js` 的失败单测被修复或与新期望对齐

### 3. 时序页双层结构最小版

第一批只落地最小可用版，不做复杂筛选和大视觉改造。

做法：

1. 保留右侧时间轴混排
2. 左侧改为弱分组结构
3. 每个条目增加来源标签
4. `daily_plan` 显示所属目标
5. 左右点击定位互通

验收标准：

1. 用户能一眼看出哪些项来自所向
2. 用户能分辨普通时序任务和习惯项
3. 左侧不再是一份来源不明的混排清单

### 4. 时序页移动端任务流收口

做法：

1. 合并 `Quick Add` 与完整模式，避免双流割裂
2. 编辑入口改成显式可见
3. 删除前统一确认
4. 保存中有 loading，成功和失败有明确反馈

要求：

1. 手机端不再依赖双击和 hover 才能编辑
2. 快速新增不再把错误默认时间悄悄写进系统

### 5. `Habits` 分端页面壳拆分

第一批正式采用方案 B。

结构要求：

1. `/habits` 路由保持不变
2. 入口层只负责端分发，不承担复杂业务逻辑
3. 桌面端页面壳组织原有高密度布局
4. 移动端页面壳独立组织内容可达路径
5. 共享逻辑统一下沉到 composable / store / `db`

硬约束：

1. 不允许把一份业务逻辑复制到两个页面壳中
2. 不允许移动端继续依赖桌面侧边栏作为唯一可见内容

### 6. `Direction` 分端页面壳拆分

第一批正式采用方案 B。

结构要求：

1. `/direction` 路由保持不变
2. 入口层按端分发
3. 桌面端保留桌面工作台式布局
4. 移动端单独组织“主内容优先”的可用结构
5. 请求、缓存、写操作逻辑继续共享

硬约束：

1. 不允许在同一页面壳里继续堆积大量 `isMobile` 条件分支
2. 不允许移动端继续处于“只剩侧边栏”的状态

### 7. 高频写操作反馈统一

第一批覆盖：

1. 任务新增 / 编辑 / 删除
2. 习惯打卡 / 编辑
3. 所向目标相关高频写操作
4. 总结保存 / 删除

统一要求：

1. 提交中有 loading
2. 成功有明确成功反馈
3. 失败有用户可理解的错误反馈
4. 危险操作有确认

### 8. Year 页数据修正

第一批必须修复 Year 页依赖 `habit.habit_logs`，但服务层未提供该结构的问题。

允许两种实现：

1. 服务层新增带日志的查询
2. 页面层并行拉 `habits + habit_logs` 再组装

约束：

1. 必须形成明确、可测试的数据契约
2. 不允许继续依赖“页面猜测服务返回结构”

## 第二批设计概要

第二批不在第一批内实现，但需要为后续设计预留方向。

1. `时序 / 所向 / 总结` 的信息架构解释统一

3. 通知权限延迟触发
4. 习惯页局部 patch、日志分页
5. 页面级单例状态回收

## 第三批设计概要

范围包括：

2. Summary 从执行数据发起的闭环
3. 视觉语法统一
4. Direction 多表写入事务化
5. 服务层用户隔离双保险

## 测试与验收

### 第一批必须覆盖

1. `Direction` 请求与缓存相关单测
2. 时序页执行项 view model 单测
3. 移动端任务新增 / 编辑核心交互测试
4. Year 页数据计算测试
5. 高频写操作反馈的最少行为测试

1. 用户能否在时序页区分来自所向的项、普通任务和习惯项？
2. 用户能否在移动端自然完成新增、编辑、删除？
3. 所向页切换目标时是否仍有重复请求和旧状态残留？

5. 年视图是否能正确展示习惯完成数据？

## 风险与约束

### 风险 1：第一批范围膨胀

控制方式：

1. 不碰一级命名
2. 不碰收件箱
3. 不碰 Summary 闭环增强
4. 不做视觉系统统一大改
5. `Habits` 与 `Direction` 的移动端只做到“可用”，不在第一批追求最终最佳体验

### 风险 2：`db` 合并牵连面过大

控制方式：

1. 只统一入口和错误契约
2. 不在第一批同时完成所有 repository 抽象升级
3. 页面先完成调用收口，再逐步增强数据层能力

### 风险 3：时序页左右两套结构脱节

控制方式：

1. 强制单一 view model
2. 强制左右互相定位
3. 不允许左右使用不同来源标签口径

### 风险 4：分端页面壳拆分后出现逻辑复制

控制方式：

1. 页面壳只组织布局和端特定交互
2. 数据获取、写操作、状态变更全部留在共享层
3. 在实现计划中明确列出共享逻辑文件与端特定页面壳文件的写入边界

## 最终结论

这次治理的核心不是“把问题列表逐条修掉”，而是先把最关键的一条业务链拉直：

`所向承诺项 -> 时序今日执行 -> 移动端可达与可操作 -> 单一 db 稳定支撑`

同时，`Habits` 与 `Direction` 的移动端已经被确认存在结构性不可用问题，因此第一批必须通过“同一路由下拆成不同页面壳”的方式把端形态边界拉开。只要第一批把这条链跑顺，第二批和第三批就能在清晰边界上继续演进；如果第一批仍然停留在零散修补，后续所有优化都会再次被状态边界和模型混乱吞掉。
