# Summary 旧结构保守冻结设计

## 背景

第一阶段数据模型对齐完成后，`summaries` 会承接新的统一运行时模型，但旧结构仍会暂时保留：

1. `daily_summaries` 旧表仍存在
2. `summaries.scope` 旧字段仍存在
3. 前端仓库里可能还残留对旧字段、旧表语义或旧组件状态的隐式依赖

如果在这个阶段立刻删除旧表和旧字段，一旦遗漏调用链，回退成本会很高。基于当前项目没有本地 Supabase 环境、数据库变更依赖 Supabase MCP 直接执行的现实约束，第二阶段更适合采用“保守冻结”策略，而不是激进清理。

## 目标

1. 让前端、服务层、文档全面切换到新的 `summaries` 统一模型。
2. 停止所有新代码对 `daily_summaries` 和 `summaries.scope` 的运行时依赖。
3. 保留旧表和旧字段作为短期回退锚点，不在本阶段执行破坏性删除。
4. 为后续真正删除旧结构提供清晰的准入条件和检查清单。

## 非目标

1. 本阶段不删除 `daily_summaries`。
2. 本阶段不删除 `summaries.scope`。
3. 本阶段不清理数据库里的旧索引、旧策略、旧约束，除非它们已经明确阻碍新模型运行。
4. 本阶段不扩展新的总结能力，如季度总结、模板系统或统计面板。

## 设计原则

### 1. 先停用，再删除

第二阶段的核心不是“删掉旧东西”，而是“确认旧东西已经不再被使用”。因此先完成依赖切断，再进入观察期；观察期通过后，删除才进入候选范围。

### 2. 以运行路径为判断标准

是否可以删除旧结构，不以“理论上应该不用了”为准，而以以下事实为准：

1. 前端新增、编辑、删除、列表展示全部只走新 `summaries`
2. 仓库内搜索不到运行时代码对 `daily_summaries` 或 `scope` 的直接依赖
3. Supabase MCP 查到的线上结构和文档记录一致

### 3. 回退优先于整洁

在没有本地数据库环境的前提下，线上数据库变更的试错成本高于仓库代码清理成本。因此本阶段优先保留回退空间，而不是追求数据库层的立即整洁。

## 范围

第二阶段保守冻结包含 4 个方面。

### 1. 前端运行路径冻结

需要把所有总结相关的运行路径统一到新模型：

1. 页面只消费 `SummaryRecord`
2. 服务层只读写 `summaries`
3. `kind + period_start + period_end + content(jsonb)` 成为唯一正式业务字段
4. `summary.date`、旧字符串 `content`、`daily_summaries` 专属字段不再进入页面逻辑

### 2. 仓库代码冻结

仓库内应完成以下冻结动作：

1. 删除或停用旧的 `daily_summaries` 数据访问函数
2. 删除或停用 UI / 业务层基于 `scope` 的旧映射逻辑，写入层兼容 bridge 可保留到线上约束解除
3. 将 `summaryTabToKind`、adapter、period builder 作为唯一入口
4. 在文档中标记 `daily_summaries` 和 `summaries.scope` 为“废弃但保留”

这里的“删除或停用”指仓库中的业务代码路径，不等于删除数据库对象。

### 3. 数据库对象冻结

数据库侧采取“保留但不再依赖”的策略：

1. `daily_summaries` 保留，不接新写入
2. `summaries.scope` 在运行时 UI / 业务层不再作为依赖项；数据库写入层当前仍保留兼容 bridge，对外统一收口到新模型，直到线上约束解除
3. 新增数据全部进入 `summaries.kind`
4. 如果需要兼容旧数据核对，只通过只读校验 SQL 使用旧对象

### 4. 文档冻结

以下文档需要同步更新：

1. `database/current-structure.md`
2. `database/README.md`
3. 总结模块 spec / plan 文档

文档必须明确写出：

1. 当前正式模型是 `summaries`
2. `daily_summaries` 为废弃保留表
3. `summaries.scope` 为运行时废弃保留字段，但写入层当前仍有兼容 bridge
4. 后续删除必须满足额外准入条件

## 冻结后的目标状态

第二阶段完成后，目标状态如下：

### 数据库

`summaries` 为唯一正式运行表，至少承担：

1. `daily`
2. `weekly`
3. `monthly`
4. `yearly`

`daily_summaries` 和 `summaries.scope` 仍然存在，但不再参与正式运行链路。
其中 `summaries.scope` 仅在写入层保留兼容 bridge，用于满足线上约束，UI / 业务层不应再直接依赖它。

### 前端

总结页的以下动作全部走新链路：

1. 列表加载
2. 新建
3. 编辑
4. 删除
5. 标题与日期格式化

### 服务层

`src/services/db/summaries.js` 成为唯一 summary 数据入口，不再同时维护双表读写。

### 文档

仓库文档能够准确回答下面两个问题：

1. 现在正式用哪张表？
2. 为什么旧表和旧字段还在？

## 准入条件

只有满足以下条件，第二阶段保守冻结才算完成：

1. 代码层已经切断对 `daily_summaries` 的运行时依赖
2. 代码层已经切断 UI / 业务层对 `summaries.scope` 的运行时依赖，写入层兼容 bridge 仍可存在
3. Summary 页面核心流程已验证：
   - 日总结新增
   - 日总结编辑
   - 周/月/年总结新增
   - 列表展示与日期显示
   - 删除
4. 仓库全文检索确认无残留关键依赖
5. MCP 校验确认线上 `summaries` 结构与当前文档一致

## 验证方式

### 1. 代码检索验证

需要对仓库执行至少以下检索：

```text
rg -n "daily_summaries|today_did|today_issue|tomorrow_fix|scope\\b" src database docs
```

预期结果：

1. `src/` 下不再出现运行时代码依赖
2. `database/` 和 `docs/` 中只允许出现“废弃保留”说明

### 2. 页面行为验证

至少验证：

1. `daily` 表单可以保存 `done / improve / tomorrow / mood`
2. `weekly/monthly/yearly` 表单可以保存 `title / text / mood`
3. 左侧列表能按 `kind` 正确显示标题和日期
4. 删除后列表与详情态同步更新

### 3. Supabase MCP 验证

至少验证：

1. `summaries` 包含 `kind / period_start / period_end / updated_at`
2. `content` 为 `jsonb`
3. `daily_summaries` 仍存在
4. `summaries.scope` 仍存在，且当前写入层兼容 bridge 仍在

这个阶段的验证目标不是确认“旧结构已删”，而是确认“旧结构已不再被依赖”。

## 风险与应对

### 风险 1：代码表面切换了，但仍有隐式旧依赖

例如：

1. 某个 sidebar 仍读取 `summary.date`
2. 某个表单仍把 `content` 当字符串处理
3. 某个删除逻辑仍分支到旧服务

应对：

1. 用 adapter 收敛数据入口
2. 用 `rg` 全仓库检索旧字段
3. 用组件级测试覆盖标题、日期、payload 形状

### 风险 2：过早删除旧结构导致线上无法快速回退

应对：

1. 第二阶段禁止破坏性删除
2. 回退时优先回退前端代码路径，而不是再做数据库结构恢复

### 风险 3：文档与实际状态再次漂移

应对：

1. 每次通过 MCP 改库后立即更新 `database/current-structure.md`
2. 在文档里标出“正式模型”和“废弃保留对象”

## 第三阶段触发条件

只有在下面条件满足后，才允许开始真正的删除阶段：

1. 第二阶段保守冻结已完成
2. 新 summary 模型线上稳定运行一段观察期
3. 用户确认不再需要以旧表/旧字段做人工核对或回退锚点
4. 删除动作已经写成单独的 spec 和执行计划

## 第三阶段候选删除项

这部分只做记录，不属于本次执行范围。

候选删除项包括：

1. `daily_summaries` 表
2. `summaries.scope` 字段
3. 旧的 summary 双表读写代码
4. 旧字段命名兼容逻辑

## 实施建议

推荐将后续工作拆成两个连续闭环：

### 闭环 A：完成冻结

1. 前端与服务层全部切到新 `summaries`
2. 文档标记旧结构为废弃保留
3. 完成代码检索、页面验证、MCP 校验

### 闭环 B：是否删除再决策

1. 观察线上使用情况
2. 确认没有回退需求
3. 再单独设计破坏性删除方案

## 结论

第二阶段采用“保守冻结”后，项目可以在不承担高回退成本的前提下，把总结模块的正式运行路径完整迁到新模型上。

这能同时满足两个目标：

1. 新模型开始真正成为唯一业务真相源
2. 旧数据库对象继续作为短期安全垫存在

等冻结完成并稳定后，再开启单独的第三阶段删除方案，会比现在直接删旧结构稳得多。
