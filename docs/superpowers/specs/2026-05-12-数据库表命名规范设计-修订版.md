## 背景

当前 Supabase `public` schema 中存在 9 张业务表和 2 张备份表。表名混用了页面名、通用名和遗留名：

- `plans`、`monthly_plans`、`daily_plans` 属于目标拆解链，但名字像全局计划系统。
- `tasks` 承载用户直接创建的任务，可进入日时间轴、月视图和总结预填。
- `daily_report_views` 的 `views` 与数据库 VIEW 概念混淆，实际是查看记录表。
- `plans_category` 单复数不统一，未表达属于目标分类。

本设计只定义目标命名与迁移影响范围，不直接执行线上表结构变更。

## 命名规则

1. 顶层业务实体主表使用单数名词：`goal`、`task`、`habit`、`summary`。
2. 子表使用父实体单数前缀，表达从属关系：`goal_months`、`goal_days`。
3. 行为日志表使用模块前缀加行为语义：`daily_report_log`。
4. 遗留表使用模块前缀加 `legacy`，避免误认为正式读写模型。
5. 不把路由名强行写进表名。`direction`、`day`、`summary` 是页面模块，数据库表表达业务领域。

## 迁移策略

采用两阶段迁移，降低一次性改名风险。

### 第一阶段：代码层对齐

1. 新增 `services/db/tables.js`，集中定义表名常量
2. 服务层文件改名，保留兼容导出以避免大面积同时修改
3. 更新所有 composables 中的引用路径
4. 移除 `summaryAdapters.js` 的 `scope` 兼容代码
5. 更新 `database.js` 导入/导出
6. 更新 `database/current-structure.md`
7. 运行 `pnpm vue-tsc` 和 `pnpm build` 验证

### 第二阶段：数据库改名

按模块分批执行，每张表改名流程：

1. `create table backup_xxx_YYYYMMDD as select * from xxx`
2. `alter table xxx rename to yyy`
3. 重建外键约束（同步改名 `plan_id` → `goal_id` 等）
4. 重建 RLS policies
5. 重建 indexes
6. 确认无报错后进入下一批

**分批顺序：**

| 批次 | 表 | 风险 |
|------|----|------|
| 1 | `habits` → `habit` | 低，仅 27 行 |
| 2 | `summaries` → `summary` | 低，仅 3 行 |
| 3 | `daily_report_views` → `daily_report_log` | 低 |
| 4 | `tasks` → `task` | 高，281 行，多处引用 |
| 5 | `plans_category` → `goal_categories` | 低，7 行 |
| 6 | `plans` → `goal` + `monthly_plans` → `goal_months` + `daily_plans` → `goal_days`（含外键改名） | 中 |

## 验收标准

1. Supabase MCP 读取的真实表结构与目标表名一致
2. Direction 页面能完成目标新增、编辑、删除、月份拆解、日期拆解
3. Day 页面能加载 `task`、`goal_days`、`habit` 并完成状态切换
4. Habits 页面能创建习惯、查询打卡、创建打卡、取消打卡
5. Summary 页面能创建、读取、更新、删除正式总结
6. 日报弹窗读取状态使用 `daily_report_log`
7. `database/current-structure.md` 与线上结构同步
8. `pnpm vue-tsc` 无类型错误，`pnpm build` 无报错
