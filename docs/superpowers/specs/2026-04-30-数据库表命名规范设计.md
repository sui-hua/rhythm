# 数据库表命名规范设计

## 背景

当前 Supabase `public` schema 中存在 10 张业务表。表名同时混用了页面名、通用名、遗留名和行为名，导致模块归属不清晰：

- `plans`、`monthly_plans`、`daily_plans` 实际属于目标拆解链，但名字像全局计划系统。
- `tasks` 实际承载用户直接创建的任务，可进入日时间轴、月视图和总结预填。
- `daily_summaries` 是旧版日总结表，但表名仍像正式数据模型。
- `daily_report_views` 实际是“每日总结/日报弹窗读取记录”表，不是数据库 view。
- `plans_category` 单复数不统一，并且未表达它属于目标分类。

本设计只定义目标命名与迁移影响范围，不直接执行线上表结构变更。

## 命名规则

1. 顶层业务实体主表使用单数名词：`goal`、`task`、`habit`、`summary`。
2. 子表使用父实体单数前缀，表达从属关系：`goal_months`、`goal_days`、`habit_checkins`。
3. 遗留表使用模块前缀加 `legacy`，避免误认为正式读写模型。
4. 行为状态表使用模块前缀加行为语义，避免和正文记录表混淆。
5. 不把路由名强行写进表名。`direction`、`day`、`summary` 是页面模块，数据库表表达业务领域。

## 目标表名

| 当前表名 | 目标表名 | 模块归属 | 说明 |
|---|---|---|---|
| `plans_category` | `goal_categories` | Direction / Goals | 目标分类表，供目标列表分组和新增目标弹窗使用 |
| `plans` | `goal` | Direction / Goals | 长期目标主表，当前代码中的 `selectedGoal`、`handleAddGoal` 对应此表 |
| `monthly_plans` | `goal_months` | Direction / Goals | 目标在月份维度的拆解节点，从属于 `goal` |
| `daily_plans` | `goal_days` | Direction / Goals | 目标在日期维度的拆解节点，从属于 `goal_months`，在 Day 页作为“今日计划”执行项展示 |
| `tasks` | `task` | Tasks | 用户直接创建的任务，包含时间轴任务、顺延任务、番茄钟计时字段，并兼容后续 inbox 能力 |
| `habits` | `habit` | Habits | 习惯定义表，使用单数主表名，与其他核心实体保持一致 |
| `habit_logs` | `habit_checkins` | Habits | 习惯打卡记录表，`checkins` 比 `logs` 更贴合创建/取消打卡语义 |
| `summaries` | `summary` | Summary | 正式总结主表，承载日 / 周 / 月 / 年总结 |
| `daily_summaries` | `summary_daily_legacy` | Summary | 旧版日总结遗留表，仅用于迁移、回退或审计 |
| `daily_report_views` | `summary_daily_report_reads` | Summary | 每日总结/日报弹窗读取记录表，记录某用户某日是否已查看 |

## 领域分组

```text
Goals:
  goal_categories
  goal
  goal_months
  goal_days

Tasks:
  task

Habits:
  habit
  habit_checkins

Summary:
  summary
  summary_daily_legacy
  summary_daily_report_reads
```

## 关系调整

目标拆解链迁移后保持原有层级：

```text
goal_categories
  └─ goal
      └─ goal_months
          └─ goal_days
```

对应外键调整：

- `goal.category_id → goal_categories.id`
- `goal_months.goal_id → goal.id`
- `goal_days.goal_month_id → goal_months.id`
- `habit_checkins.habit_id → habit.id`
- `habit_checkins.user_id → auth.users.id`
- `summary_daily_report_reads.user_id → auth.users.id`

## 字段命名调整

第一阶段只调整前端服务层命名，不改线上字段。第二阶段执行数据库真实改名时，同步调整目标链外键字段，避免新表名继续携带旧的 `plan` 语义。

字段层面的明确改动：

- `goal_months.plan_id` 应改为 `goal_id`。
- `goal_days.monthly_plan_id` 应改为 `goal_month_id`。

字段层面的遗留问题：

- `habit.archived` 与 `habit.is_archived` 应收敛为单一字段。
- `summary.scope` 是兼容字段，正式模型已收敛到 `kind + period_start + period_end`。

## 迁移策略

采用两阶段迁移，降低一次性改名风险。

### 第一阶段：代码服务层对齐

1. 新增表名常量映射，让服务层集中引用表名。
2. 将服务导出名逐步从 `plans`、`monthlyPlans`、`dailyPlans` 过渡到 `goal`、`goalMonths`、`goalDays`。
3. 保留兼容导出，避免一次性修改所有调用点。
4. 更新测试，覆盖 Goal、Habit、Summary 关键查询路径。

### 第二阶段：数据库真实改名

1. 通过 Supabase MCP 确认最新线上表结构、外键、索引、RLS policy 和 RPC。
2. 创建迁移 SQL，执行 `alter table ... rename to ...`。
3. 同步修改外键字段名：`plan_id → goal_id`，`monthly_plan_id → goal_month_id`。
4. 重建或重命名相关 constraint、index、policy、RPC。
5. 更新前端服务层表名常量。
6. 运行构建、单元测试和关键页面手测。
7. 更新 `database/current-structure.md`。

## 验收标准

1. Supabase MCP 读取的真实表结构与目标表名一致。
2. Direction 页面能完成目标新增、编辑、删除、月份拆解、日期拆解。
3. Day 页面能加载 `task`、`goal_days`、`habit` 并完成状态切换。
4. Habits 页面能创建习惯、查询打卡、创建打卡、取消打卡。
5. Summary 页面能创建、读取、更新、删除正式总结。
6. 每日总结/日报弹窗读取状态使用 `summary_daily_report_reads`。
7. `database/current-structure.md` 与线上结构同步。
