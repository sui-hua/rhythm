# Day 视图改进设计方案

## 概述

针对高频使用者反馈，对 Day 视图进行三项改进，提升日常任务添加、完成和通知的效率。

## 优先级

1. 快速添加模式（移动端）
2. 移动端左滑完成任务
3. 精确到分钟通知（Service Worker）

---

## 1. 快速添加模式

### 背景

用户主要在 Day 视图进行「添加任务」和「完成任务」操作。当前添加流程需要填写：标题、时间、时长、描述、分类共 5 个字段，对高频快速添加场景过于繁琐。

### 设计方案

**移动端底部浮动按钮**
- 位于右下角，圆形 + 按钮（现有添加入口）
- 点击后弹出「快速添加」模式

**极简输入模式**
- 全屏输入框，只要求输入标题
- 时间默认：上个任务结束时间或 08:00
- 时长默认：30 分钟
- 分类默认：工作
- 键盘自动聚焦，回车直接创建

**一键切换完整模式**
- 输入框下方「更多选项」按钮
- 点击展开完整表单（时间、时长、描述、分类）

**交互流程**
```
点击 [+] → 弹出快速输入 → 输入标题 → 回车创建 → 刷新列表
```

### 技术实现

- 新增 `QuickAddDrawer` 组件（移动端）
- 复用 `useAddEventForm` 的 `lastUsedTime` 逻辑
- 创建后自动聚焦回输入框，支持连续快速添加

## 2. 移动端左滑完成任务

### 背景

移动端 Checkbox 点击区域小，任务完成操作效率低。需要更符合移动端习惯的快捷操作。

### 设计方案

**左侧滑动交互**
- 任务列表项支持左滑手势
- 滑出区域显示完成按钮（绿色背景 + 勾选图标）
- 滑动距离超过 50% 自动触发完成
- 未超过阈值则回弹

**视觉反馈**
- 滑动时显示半透明绿色区域
- 完成按钮有触感反馈（vibration API）
- 完成成功后播放成功音效

**兼容性**
- 同时保留 Checkbox 点击方式
- 左滑仅在移动端启用（`useMobile` 检测）

### 技术实现

- 新增 `SwipeableTaskItem` 封装组件
- 使用 Touch Events 检测滑动方向和距离
- 复用 `useDayData.handleToggleComplete`

---

## 3. 精确到分钟通知（Service Worker）

### 背景

当前通知只在整点触发，无法满足精确时间提醒需求（如 9:30 的任务在 9:30 提醒）。

### 设计方案

**Service Worker 后台运行**
- 注册独立 Service Worker 脚本
- 页面关闭时也能触发系统通知

**每分钟任务检查**
- 从 IndexedDB 读取当天任务时间表
- 每分钟检测是否有任务到达指定分钟数
- 避免重复通知（记录已通知任务 ID）

**通知内容**
- 标题：任务名称
- 内容：任务描述或「点击查看详情」
- 点击通知聚焦页面

### 技术实现

- 新增 `sw.js` Service Worker 文件
- IndexedDB 缓存当日任务时间表
- Notification API + Service Worker 组合
- 复用 `useNotifications` 的权限管理和通知逻辑

### 降级方案

若 Service Worker 不可用，回退到原 `useNotifications` 的轮询方案。

---

## 实施顺序

1. **快速添加模式** → 最快提升高频操作效率
2. **移动端左滑完成** → 优化移动端核心交互
3. **精确到分钟通知** → 需要 Service Worker 基础设施
