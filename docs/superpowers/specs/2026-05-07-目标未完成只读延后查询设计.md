# 目标未完成只读延后查询设计

## 概述

Direction 目标的日计划目前只会在原始 `day` 当天被 Day 页面查出。用户希望未完成的目标任务在后续日期继续可见，但不希望系统改写原始承诺日期。

本方案将“自动延后”收敛为**只读延后查询**：不修改 `daily_plans.day`，只在 Day 页面查询时，按目标级配置补查过去若干天内未完成的目标日计划，并把它们展示为当天仍需处理的事项。

---

## 设计目标

- 每个目标单独配置是否启用未完成延后展示
- 配置默认关闭，旧数据不受影响
- Day 页面可以把历史未完成目标任务继续展示出来
- 保留 `daily_plans.day` 的原始日期，用于追溯真实承诺时间
- 完成历史未完成项时，仍然更新原记录，而不是生成副本或迁移日期

## 非目标

- 不修改 `daily_plans.day`
- 不新增“把任务真正顺延到明天”的写入逻辑
- 不为 `monthly_plans` 或 `daily_plans` 单独保存这份规则
- 不把普通 `tasks` 的顺延逻辑并入本次改动
- 不引入新的配置表
- 当前阶段不直接通过 MCP 修改线上 Supabase schema，只在文档中明确字段方案

---

## 方案选择

### 备选方案

1. 在 `plans` 上新增目标级字段
2. 新建 `plan_settings` 一类配置表
3. 仅保存在前端本地偏好

### 选定方案

在 `plans` 表新增 `carry_over_lookback_days` 字段，表示该目标在 Day 页面中向前补查多少天未完成的 `daily_plans`。

选择该方案的原因：

- 配置和目标本身同生命周期，语义最直接
- 删除目标时无需额外清理配置表
- 当前只需要一个规则字段，新建配置表会引入额外复杂度
- 本地偏好无法跨设备同步，也不适合作为目标级业务规则

---

## 数据模型

### `plans` 新增字段

- 字段名：`carry_over_lookback_days`
- 类型：整数
- 默认值：`0`
- 语义：
  - `0`：关闭未完成延后展示
  - `> 0`：Day 页面额外补查该目标过去 N 天内未完成的日计划

### 字段命名说明

数据库字段保留 `carry_over_lookback_days`，因为它明确表达了“向前回看多少天以补查历史未完成项”的技术语义。

前端展示文案不直接暴露英文名，使用中文产品文案，例如：

- `未完成延后展示`
- `延后展示天数`

### 当前阶段约束

本次 brainstorming/spec 阶段只把字段方案写入文档，不直接通过 MCP 对线上 Supabase 执行 `ALTER TABLE`。真实改库动作放到实现阶段执行，并且直接通过 Supabase MCP 落库，再和前端读写链路一起验证。

## 查询行为设计

### 当前行为

Day 页面当前通过 `db.dailyPlans.listByDate(startOfDay)` 只查询某一天 `day = 当天日期` 的目标日计划。

### 新行为

Day 页面查询某一天时，目标日计划来源拆成两部分：

1. 原始当天日计划：
   - `day = 目标日期`
2. 历史未完成补查日计划：
   - `day < 目标日期`
   - `day >= 目标日期 - carry_over_lookback_days`
   - `status = 0`
   - 关联目标 `plans.carry_over_lookback_days > 0`

最终把两部分结果合并为当天展示列表。

### 去重规则

同一条 `daily_plans` 记录只展示一次。

当天原始日计划与历史补查日计划天然不会重叠，因为两者 `day` 条件不同，但合并阶段仍保持按 `daily_plans.id` 去重，避免后续查询扩展时重复。

### 排序规则

展示顺序延续现有 Day 页面时间线规则：

- 优先按 `task_time` 参与时间线排序
- 无需因为“历史未完成”改变其在时间线中的展示模型

如果需要在 UI 上区分来源，只增加标记，不改变排序基准。

## Day 页面展示规则

### 展示原则

- 历史未完成目标任务在今天可见，但仍保留原始计划日期
- 用户完成它时，更新的是原 `daily_plans` 记录的 `status`
- 用户看到的是“今天仍需处理”，不是“系统已经替你改到今天”

### 建议的界面信息

为了避免用户误解任务日期被系统改写，历史补查项在 Day 页面需要有清晰来源提示。推荐显示：

- `原计划 5 月 4 日`

或

- `5 月 4 日未完成`

首轮不强制加入新的分组区域，只要在现有任务项内能标清原始日期即可。

## 配置入口设计

### 入口位置

配置放在 Direction 的 Add/Edit 目标弹窗中，和目标自己的时间、时长配置并列。

涉及入口：

- `src/views/direction/components/AddGoalModal.vue`
- `src/views/direction/composables/useDirectionGoals.js`

### 交互规则

- 默认关闭
- 用户开启后，填写一个正整数天数
- 用户关闭后，保存值回到 `0`

### 文案建议

- 开关：`未完成延后展示`
- 数值输入：`延后展示天数`
- 辅助说明：`开启后，Day 页面会额外显示该目标过去若干天未完成的日计划，但不会修改原计划日期。`

### 默认值

- 新建目标：默认关闭，对应 `carry_over_lookback_days = 0`
- 编辑旧目标：若数据库中无值，按 `0` 处理

## 关键实现约束

1. `carry_over_lookback_days` 只定义“补查窗口”，不定义任何写入迁移行为。
2. 历史未完成项完成后，只更新原记录 `status`，不复制记录。
3. 查询条件必须限制在所属目标配置开启的前提下，不能把所有历史未完成目标任务都带到当天。
4. UI 必须能看出这是历史未完成项，避免用户误读为系统篡改日期。
5. 关闭配置后，该目标历史未完成项应立即停止在后续日期展示。

## 风险与注意点

### 1. 查询复杂度上升

`daily_plans.listByDate()` 目前只按单日查询。引入目标级补查后，查询需要关联 `monthly_plans -> plans` 并附带窗口过滤，复杂度明显上升。

### 2. Day 页面认知负担上升

如果历史未完成项没有明确标记，用户会误以为任务的原始日期被改了。

### 3. 历史积压过多时的展示密度

虽然目标级配置默认关闭，但一旦某些目标开启较大天数，Day 页面可能出现较多历史积压项，因此需要保证视觉上能区分“今天的计划”和“历史未完成”。

---

## 验收标准

### 数据行为

- 新建目标时，`carry_over_lookback_days` 默认为 `0`
- 编辑目标时，可以单独开启或关闭该配置
- 关闭时，数据库中保存为 `0`

### 查询行为

- 当目标配置为 `0` 时，Day 页面只显示当天原始 `daily_plans`
- 当目标配置为 `N > 0` 时，Day 页面会额外显示该目标过去 N 天未完成的 `daily_plans`
- 补查结果不会修改任何 `daily_plans.day`

### 交互行为

- 用户完成历史未完成项后，原记录状态更新为完成
- Day 页面能明确展示历史未完成项的原始日期
- 用户不会把该行为误解为“任务日期被系统改写”
