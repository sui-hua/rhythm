# 项目审查报告

> 审查日期：2026-06-06
> 审查范围：前端全模块（7 个视图模块）、数据服务层、状态管理、类型系统、构建配置

---

## 一、项目目标理解

- **项目核心目标**：rhythm 是一个个人效率管理工具，核心能力为"以日为单位管理时间、目标、习惯"，帮助用户建立可追踪的自律节奏
- **目标用户**：追求自我管理的个人用户（非团队协作工具）
- **核心使用场景**：每日时间轴规划 → 目标拆解到日 → 习惯打卡 → 日终复盘
- **本次审查范围**：前端全模块（7 个视图模块）、数据服务层、状态管理、类型系统、构建配置

## 二、最影响用户体验的 3 个问题

### 1. 类型系统全面失效，编译期安全形同虚设

- **优先级**：高
- **所属维度**：代码质量 / 前端架构
- **问题**：`createBase<T>` 工厂函数的 `create`、`update`、`query` 方法内部全部使用 `as any` 绕过 Supabase 类型约束；`getById` 返回 `Promise<T>` 不反映"未找到"场景；`Goal`、`Task` 等接口缺失数据库实际存在的 `status`、`year`、`user_id` 等字段；`GoalDay.status` 定义为宽泛 `string` 而非联合类型
- **原因**：Supabase JS SDK 的 Insert/Update 类型与前端定义的实体类型不兼容，开发时选择了用 `as any` 快速绕过而非定义精确的 Payload 类型
- **修改建议**：为每个表定义精确的 Insert/Update 类型（可从 `supabase gen types` 生成）；补全 `Goal`、`Task` 接口缺失字段；将 `GoalDay.status` 收窄为联合类型；`getById` 改用 `maybeSingle()` 返回 `T | null`
- **预期收益**：编译期捕获 80% 以上的字段拼写错误和类型不匹配，减少运行时 bug

### 2. 删除操作无二次确认，误操作不可逆

- **优先级**：高
- **所属维度**：UX / 产品
- **问题**：`AddEventModal`（删除任务）、`EditHabitModal`（删除习惯）、`AddGoalModal`（删除目标）三处删除操作均直接执行，无确认对话框。习惯和目标属于长期数据资产，误删后果严重
- **原因**：开发阶段追求操作效率，未对危险操作设计防护流程
- **修改建议**：所有删除操作前弹出确认对话框，明确告知"此操作不可恢复"；项目已引入 Reka UI 的 Dialog 组件，可复用
- **预期收益**：消除误删风险，建立用户对数据安全的信任感

### 3. 番茄钟无法暂停/放弃，用户被困在流程中

- **优先级**：高
- **所属维度**：UX
- **问题**：`PomodoroTimerModal` 只有"完成并记录"按钮，没有暂停、放弃或关闭计时器的出口；弹窗关闭按钮被 CSS 隐藏（`[&>button]:hidden`），用户一旦开始计时无法退出
- **原因**：功能实现时只考虑了"完成"的理想路径，未设计中断/异常路径
- **修改建议**：增加"暂停"和"放弃本次"按钮，恢复弹窗默认关闭按钮，关闭时提示"计时未完成，确定放弃？"
- **预期收益**：消除用户被"困住"的挫败感，符合番茄钟工作法的实际使用习惯

## 三、详细问题清单

### 产品逻辑

#### 问题 1：日报功能与总结模块功能重叠

- **优先级**：中
- **问题**：`DailyReportModal` 提供任务完成统计和顺延功能，`Summary` 模块的 `DailySummaryForm` 也展示"今日数据概览"并要求填写成就/改进/计划。两者都在做"日终复盘"，入口分散在不同模块
- **原因**：两个功能在不同时期独立开发，未做产品层面的统一规划
- **修改建议**：二选一——要么将日报弹窗的数据概览能力合并到 Summary 模块，要么取消 Summary 的日总结功能由日报弹窗承担

#### 问题 2：Month/Year 视图功能单薄，缺乏实际价值

- **优先级**：中
- **问题**：`month/index.vue` 和 `year/index.vue` 仅有日历网格展示，没有聚合任何业务数据（任务完成率、习惯打卡率、目标进度）
- **原因**：功能处于 MVP 阶段，数据聚合逻辑尚未实现
- **修改建议**：补充关键指标聚合（如每日完成率热力图），或暂时隐藏这两个路由避免"产品没做完"的印象

#### 问题 3：Direction 模块三级级联过深，操作成本高

- **优先级**：中
- **问题**：用户需要 选择目标 → 选择月份 → 查看/编辑日计划，至少三次点击才能看到具体执行项
- **原因**：数据模型按 goal → goal_months → goal_days 三级拆分，UI 直接映射了数据结构
- **修改建议**：提供"今日目标"快捷视图，直接从 `goal_days` 表拉取当天待办，与 Day 时间轴并列展示

#### 问题 4：Summary 模块缺乏引导触发机制

- **优先级**：低
- **问题**：总结是低频但高价值的行为，当前空状态只显示"无总结"，没有时间维度的提示
- **修改建议**：在 Day 页面日报弹窗确认后自动跳转到 Summary 的日总结表单；侧边栏对未写总结的时段做视觉标记

### 用户体验（UX）

#### 问题 1：单击/双击操作不可发现

- **优先级**：高
- **问题**：时间轴任务卡片使用"单击选中/双击编辑"，但双击在 Web 应用中极为罕见，无任何视觉提示
- **修改建议**：改为单击卡片打开编辑（最常用操作），或在首次使用时通过 tooltip 引导

#### 问题 2：习惯打卡无撤销机制，误触率高

- **优先级**：中
- **问题**：日历中点击任意日期格子直接触发打卡/取消打卡，无确认步骤；toast 提示没有撤销按钮
- **修改建议**：toast 中增加"撤销"按钮（类似 Gmail undo），或对取消打卡操作增加确认

#### 问题 3：登录页缺少注册入口

- **优先级**：中
- **问题**：登录页只有邮箱+密码表单和演示账号入口，无注册链接。新用户无法自助注册
- **修改建议**：增加"注册" tab 或将按钮文案改为"登录/注册"，利用 Supabase 的 signUp 能力

#### 问题 4：日报弹窗关闭即丢失

- **优先级**：低
- **问题**：用户点击弹窗外部直接关闭日报，无"稍后提醒"选项
- **修改建议**：禁用点击外部关闭，或在关闭时提供"明天再提醒"选项

### UI 设计

#### 问题 1：大量硬编码色值绕过设计令牌

- **优先级**：高
- **问题**：`TaskItem.vue` 大量使用 `bg-zinc-50/80`、`border-zinc-300`、`text-zinc-400`；`Navbar.vue` 激活态使用 `bg-black text-white`，暗色模式下黑底黑字不可见
- **原因**：开发时未建立严格的设计令牌使用规范
- **修改建议**：统一替换为 `bg-muted`、`border-border`、`text-muted-foreground` 等语义 class；激活态改为 `bg-primary text-primary-foreground`

#### 问题 2：弹窗底部按钮风格不统一

- **优先级**：低
- **问题**：`AddEventModal` 中删除按钮使用原生 `<button>` + 手写样式，其他按钮使用 `<Button>` 组件，同一操作区内混用
- **修改建议**：删除操作也使用 `<Button variant="link">` 统一

#### 问题 3：Dialog 遮罩层过重

- **优先级**：低
- **问题**：`DialogOverlay` 使用 `bg-black/80`，与其他元素的 `backdrop-blur-xl` 毛玻璃风格不协调
- **修改建议**：改为 `bg-black/40 backdrop-blur-sm`

### 性能

#### 问题 1：lucide-vue-next 整包打入首屏

- **优先级**：高
- **问题**：`vite.config.ts` 将 `lucide-vue-next`（1000+ 图标）整体归入 `ui-vendor` chunk，首屏加载巨型文件
- **修改建议**：改为按需导入路径 `lucide-vue-next/icons/xxx`，或单独拆为 `icons-vendor` chunk

#### 问题 2：Day 页面无缓存，每次进入重新拉取全部数据

- **优先级**：中
- **问题**：`dayStore.fetchTasks` 无缓存机制，切换日期后返回同一天会再次发出 3 个并行请求
- **修改建议**：维护内存缓存，短时间内重复访问同一天时直接使用缓存

#### 问题 3：`goalDataStore` 使用 `reactive({})` + `delete` 操作缓存

- **优先级**：中
- **问题**：大量 `delete` 操作触发不必要的代理追踪开销
- **修改建议**：改用 `ref<Map<string, T>>()`，批量替换时直接赋值新 Map

#### 问题 4：`useHabitStats` O(habits * logs) 复杂度

- **优先级**：低
- **问题**：对每个习惯遍历全量 logs 计算今日完成状态
- **修改建议**：在数据获取层按日期预筛选，或维护 `todayCompletedIds` Set 缓存

### 前端架构

#### 问题 1：dayStore 在 store 内部调用 composable，职责边界倒置

- **优先级**：高
- **问题**：`dayStore.ts` 直接调用 `useDayActions` composable 并将其返回值作为 store 导出，违反分层约定，形成 store ↔ composable 双向耦合
- **修改建议**：将 `useDayActions` 调用移至组件层，dayStore 仅导出状态和数据方法

#### 问题 2：goalDataStore 双源数据（扁平数组 + 缓存对象）

- **优先级**：中
- **问题**：同时维护 `goalMonths` 扁平数组和 `goalMonthsCache` 映射，通过 `syncGoalMonthsToFlatList` 手动同步，是 bug 温床
- **修改建议**：废弃 `goalMonths` 扁平列表，统一使用 `goalMonthsCache`，需要扁平视图时通过 computed 派生

#### 问题 3：DateContext 接口重复定义

- **优先级**：低
- **问题**：`dayStore.ts` 和 `useDayActions.ts` 各自定义了完全相同的 `DateContext` 接口
- **修改建议**：提取到 `types/models.ts` 统一定义

#### 问题 4：modalState.ts 模块级共享 ref 存在状态泄漏风险

- **优先级**：低
- **问题**：弹窗状态定义为模块顶层 ref，无法按组件生命周期隔离
- **修改建议**：收敛到 `goalSelectionStore` 或使用 `provide/inject`

### 后端架构

#### 问题 1：`createBase` 默认 `orderField='created_at'` 对无该字段的表是陷阱

- **优先级**：中
- **问题**：`habit_logs` 表没有 `created_at` 字段，但 `createBase` 默认按此排序
- **修改建议**：移除默认值或改为必填参数

#### 问题 2：`db.rpc` 完全无类型约束

- **优先级**：中
- **问题**：`rpc(name: string, params?: Record<string, any>)` 参数和返回值全为 `any`，无法捕获 RPC 名称拼写错误
- **修改建议**：定义 RPC 名称常量枚举，为每个 RPC 提供参数/返回值类型映射

#### 问题 3：`user_id` 传递策略不统一

- **优先级**：低
- **问题**：`habit_logs` 有 `auth.uid()` 默认值无需传，但其他表的 Payload 中 `user_id` 有的可选有的必填
- **修改建议**：统一约定——有数据库默认值的字段，前端 Payload 标记为可选

#### 问题 4：`getById` 返回类型不反映"未找到"场景

- **优先级**：中
- **问题**：`.single()` 无匹配时抛出错误而非返回 null，但返回类型声明为 `Promise<T>`
- **修改建议**：改用 `.maybeSingle()` 返回 `Promise<T | null>`

### 数据架构

#### 问题 1：`habit.log()` 无幂等保护，可重复打卡

- **优先级**：高
- **问题**：直接 insert 新记录，未检查同一天是否已打卡，快速双击或网络重试导致重复记录
- **修改建议**：在 `habit_logs` 表添加唯一约束 `UNIQUE(habit_id, date(completed_at))`，或在 `log()` 方法中先查询当日记录

#### 问题 2：Goal/Task 前端接口缺失关键数据库字段

- **优先级**：高
- **问题**：`Goal` 接口缺少 `status`、`year`、`user_id`；`Task` 接口缺少 `status`（但代码中使用了 `.eq('status', 'inbox')`）
- **修改建议**：将前端接口与数据库表结构对齐

#### 问题 3：`listForDayView` 存在 N+1 风险

- **优先级**：中
- **问题**：每次调用先查 `goal` 表获取最大回溯天数，再查 `goal_days` 表，两次独立网络往返
- **修改建议**：将回溯天数缓存到 Pinia store，仅在目标数据变更时刷新

#### 问题 4：核心业务表缺乏软删除机制

- **优先级**：低
- **问题**：所有表的 `delete` 均为硬删除，`habit` 被删后 `habit_logs` 可能因外键失败或残留孤儿记录
- **修改建议**：对核心表引入 `deleted_at` 软删除字段，或确认外键的 CASCADE 行为

### 系统架构

#### 问题 1：Store 与 Composable 反向依赖

- **优先级**：高
- **问题**：`dayStore` 导入 `views/day/composables/useDayActions`，Store 属于基础设施层却依赖视图层，依赖方向混乱
- **修改建议**：将操作逻辑上提到 `src/composables/` 或内聚到 store，保持 Store → Service 单向依赖

#### 问题 2：Direction 模块 Store 碎片化

- **优先级**：中
- **问题**：三个 store + 一个 `modalState.ts` 共享 ref，高度耦合但分散管理，增加状态同步复杂度
- **修改建议**：将 `goalSelectionStore` 和 `goalBatchStore` 合并为 `goalUIStore`

#### 问题 3：错误处理缺乏用户反馈闭环

- **优先级**：中
- **问题**：`safeAction` 捕获错误后仅 `console.error`，乐观更新失败后用户无感知
- **修改建议**：在 `safeAction` 中集成 toast 通知（已引入 `vue-sonner`）

#### 问题 4：`sessionVerified` 一次验证后永不过期

- **优先级**：中
- **问题**：路由守卫使用模块级变量控制首次 session 验证，验证通过后不再检查，token 过期无法感知
- **修改建议**：结合 `onAuthStateChange` 监听 token 刷新失败，或加入 session 过期时间检查

### 代码质量 / 全栈 Review

#### 问题 1：乐观更新回滚不完整

- **优先级**：高
- **问题**：`handleToggleComplete` 中任务完成切换失败后，catch 只回滚了 `completed` 字段，未清除 `actual_end_time`
- **修改建议**：回滚时也还原 `actual_end_time`

#### 问题 2：`carryOverUncompletedTasksTo` 中 `start_time` 为 null 时产生 Invalid Date

- **优先级**：中
- **问题**：`new Date(task.start_time || '')` 当 `start_time` 为 null 时返回 Invalid Date，后续操作产生 NaN 日期写入数据库
- **修改建议**：过滤掉 `start_time` 缺失的任务或赋予默认时间

#### 问题 3：缓存命中不考虑数据过期

- **优先级**：中
- **问题**：Direction 模块仅通过缓存 key 是否存在判断是否跳过请求，多标签页/设备修改数据后缓存永不失效
- **修改建议**：增加 TTL 或提供手动刷新入口

#### 问题 4：目标切换 watcher 存在竞态

- **优先级**：中
- **问题**：快速切换目标时，前一次请求未完成新 watcher 触发，`isInitializing` 守卫失效，可能覆盖数据
- **修改建议**：增加取消令牌或版本号校验

## 四、跨维度关键问题

### 1. 类型系统全面失效

- **影响维度**：代码质量 + 前端架构 + 后端架构 + 数据架构
- **说明**：`createBase` 的 `as any`、接口缺失字段、宽泛 string 类型三个问题互为因果——Supabase 类型不兼容 → 用 any 绕过 → 接口定义偷懒 → 运行时 bug 无法在编译期捕获
- **建议优先处理方式**：从 Supabase 生成精确类型 → 补全前端接口 → 消除 `as any`

### 2. Store ↔ Composable 边界模糊

- **影响维度**：前端架构 + 系统架构 + 性能
- **说明**：dayStore 调用 composable、goalDataStore 双源数据、modalState 共享 ref 三个问题共同暴露了状态管理层缺乏统一规范。导致依赖方向混乱、数据同步风险、缓存策略不一致
- **建议优先处理方式**：制定 Store/Composable 分层规范文档，重构 dayStore 的 composable 调用模式

### 3. 危险操作缺乏防护

- **影响维度**：UX + 产品 + 数据架构
- **说明**：删除无确认、打卡无撤销、番茄钟无退出、习惯可重复打卡——四个问题共同指向"操作安全性"设计缺失
- **建议优先处理方式**：统一为所有不可逆操作增加确认/撤销机制

## 六、总结结论

**当前项目最大的问题**：类型系统的全面失效。`createBase` 工厂函数的 `as any` 像病毒一样扩散到整个数据层，使得 TypeScript 的核心价值——编译期类型安全——完全落空。这不是某一处疏忽，而是架构层面的系统性缺陷。

**当前项目最值得保留的优点**：Composable 分层设计的方向是正确的——Day、Direction、Habits 模块都有清晰的 composable 抽象，职责划分意图明确。问题在于执行层面的边界模糊，而非方向错误。

**如果只能改 3 个点**：

1. 消除 `createBase` 的 `as any`，定义精确的 Insert/Update 类型（影响面最广，修复后全链路受益）
2. 所有删除操作增加确认对话框（成本最低，用户感知最强）
3. dayStore 移除对 useDayActions 的直接调用，恢复正确的分层依赖（架构健康度的标志性修复）

## 七、犀利辣评

> 看完 rhythm 的代码，我最大的感受是：**这个项目的类型系统是个摆设，而 store 和 composable 的关系已经乱成了意大利面。**
>
> 先说类型。`createBase<T>` 里 `as any` 满天飞——`insert`、`update`、`query`，每条路径都有一行 `eslint-disable-next-line @typescript-eslint/no-explicit-any` 加一句"不兼容"的注释，像极了考试不会做题写个"略"。`BaseService<T>` 的 `getById` 返回 `Promise<T>`，意思是"一定能查到"，但数据库显然不这么想。`types/models.ts` 号称是"共享数据模型"，实际上只是个 re-export 集散地，加上一个 `AugmentedHabit` 接口——这些扩充字段全靠 composable 手动往对象上糊，没有任何运行时保证。你花了整个项目的力气给 TypeScript 服务，最后全靠 `as any` 一笔勾销，这跟不穿裤子有什么区别？
>
> 再说架构。`dayStore` 在 store 定义体内直接调 `useDayActions`——一个 store 居然反过来消费 composable，composable 里又通过 `db` 调服务层，`db` 里又是 `createBase` 返回的 `as any`。数据流是 **store → composable → service → any**，类型信息在每一层都流失一点，到数据库层已经什么都不剩了。Direction 模块更离谱：三个 store 加一个 `modalState.ts` 共享 ref，再加上 `useDirectionFetch` 里用模块级变量当锁来阻止 watcher 连环触发——这不是架构设计，这是在用创可贴补大坝。
>
> 最后说缓存策略。Direction 模块手写了一套 `reactive({})` 缓存，散落在 composable 里手动管理；Day 模块则完全没有缓存，每次 `fetchTasks` 都全量拉。同一项目两种截然不同的缓存哲学，像一个人左手用筷子右手用刀叉，还觉得自己很灵活。
>
> **一句话判决：TypeScript 画皮、composable 和 store 鸡蛋混炒、缓存各搞各的——这不是一个 Vue 3 应用，这是一个用 `.ts` 后缀伪装的 JavaScript 项目。**
