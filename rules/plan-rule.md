# 用户偏好与工作流规则

## 计划语言

**规则：** 写计划时必须使用中文。

**Why：** 统一团队沟通语言，减少计划评审和执行过程中的理解偏差。

**How to apply：**
- 设计文档、实现计划、执行清单统一使用中文编写
- 不用英文作为主叙述语言
- 必须保留英文时，仅用于代码标识、文件路径、接口名等必要上下文

## Git 提交时机

**规则：** 完成一整个功能后再提交。

**Why：** 避免把半成品、临时状态或拆得过碎的改动提交进历史，保持提交语义完整。

**How to apply：**
- 不在功能进行到一半时提交
- 只有当一个功能已经实现完整、验证完成、可以独立成立时才进行提交
- 如果任务被拆成多个步骤，优先在功能闭环完成后统一提交

## Git 提交权限

**规则：** 实现计划执行完成后，不要执行 `git commit`，最后由用户自己提交。

**Why：** 用户希望保留最终的提交权限，控制提交时机和提交信息。

**How to apply：**
- 跳过所有 `git commit` 步骤
- 代码修改完成后直接告知用户验证
- 用户确认无误后自己执行提交
