## 问题 1：服务层缺少统一的 `user_id` 查询边界，核心数据读取依赖数据库侧兜底
- 标题：查询层未显式按用户隔离
- 证据：
  - [src/services/db/tasks.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/db/tasks.js:21)
  - [src/services/db/plans.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/db/plans.js:21)
  - [src/services/db/monthlyPlans.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/db/monthlyPlans.js:21)
  - [src/services/db/dailyPlans.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/db/dailyPlans.js:22)
  - [src/services/db/habits.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/db/habits.js:27)
  - [src/services/db/summaries.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/db/summaries.js:44)
  - [database/current-structure.md](/Users/sifan1/Documents/project/日程/rhythm/database/current-structure.md:20)
- 问题：
  - 这些表在数据库结构里都带有 `user_id`，但前端服务层的大多数查询没有显式 `.eq('user_id', currentUserId)`。
  - 当前实现把“用户隔离”几乎完全押注在 Supabase RLS 或数据库默认行为上，服务层自身不具备边界意识。
- 原因：
  - `createBase()` 只是表级 CRUD 工厂，没有“当前用户上下文”概念。
  - 上层 composable 直接调用裸服务，缺少 repository / query policy 层来注入租户边界。
- 影响：
  - 一旦 RLS 配置缺失、放宽或未来复用到 server-side 环境，马上会出现跨用户串读。
  - 本地单测和前端行为无法验证“用户隔离”是否成立，问题会延后到线上暴露。
  - 维护者很难判断某个查询是否安全，因为约束不在代码层可见。
- 建议：
  - 在服务层增加统一的“用户作用域”封装，例如 `scopedTable('tasks')`，由该层强制注入 `user_id`。
  - 对所有读写接口建立明确约定：要么显式传 `userId`，要么只允许从 auth context 读取。
  - 把“依赖 RLS”从隐式前提改成显式防线：代码层过滤 + RLS 双保险。
- 优先级：高

## 问题 2：`safeDb` 的错误契约不一致，调用方无法可靠区分“空数据”和“请求失败”
- 标题：错误处理层吞错且返回值语义混乱
- 证据：
  - [src/services/safeDb.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/safeDb.js:72)
  - [src/services/safeDb.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/safeDb.js:99)
  - [src/views/day/composables/useDayData.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/composables/useDayData.js:78)
- 问题：
  - 标准 CRUD 失败时，`safeDb` 会 toast 后直接 `return null`。
  - 非标准方法如 `listByDate`、`listLogsByDate`、`log`、`deleteLog` 又是直接透传原始异常。
  - 这导致同一个 `safeDb` 内部同时存在两种完全不同的失败语义。
- 原因：
  - `wrapTable()` 只包装 `list/create/update/delete`，其它方法完全绕过统一异常协议。
  - 上层 composable 又普遍把返回值当作成功结果继续使用。
- 影响：
  - 调用方无法知道 `null` 是“真的没有数据”还是“请求失败”。
  - 某些地方会保留旧状态，形成“界面看起来正常，但其实写入失败”的假成功。
  - 随着服务方法增多，错误处理会越来越分裂，维护和排障成本显著上升。
- 建议：
  - 统一服务层契约：失败时全部 `throw`，toast 只放在页面层或 action 层。
  - 如果必须返回对象，建议统一为 `{ data, error }`，不要混用 `null` 与异常。
  - `wrapTable()` 应覆盖所有方法，或者废弃 `safeDb` / `db` 双入口，统一单一调用方式。
- 优先级：高

## 问题 3：多个 composable 以模块级单例保存页面数据，导致状态泄漏和实例间隐式耦合
- 标题：页面状态被提升到模块顶层，失去实例边界
- 证据：
  - [src/views/day/composables/useDayData.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/composables/useDayData.js:36)
  - [src/views/day/composables/useDailyReport.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/composables/useDailyReport.js:25)
  - [src/views/direction/composables/useDirectionState.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/composables/useDirectionState.js:36)
  - [src/views/direction/composables/useDirectionFetch.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/composables/useDirectionFetch.js:40)
- 问题：
  - `useDayData` 的 `tasks/dailyPlans/habits/habitLogs/isLoading` 是模块单例。
  - `useDailyReport` 的 `reportVisible/reportStats` 也是模块单例。
  - Direction 模块更进一步，把缓存、选择态、watch 初始化开关都做成全局共享。
- 原因：
  - 代码把“减少重复请求”和“共享状态”直接等同起来，但没有设计重置机制与实例生命周期。
  - 这里其实已经接近 store 的职责，却没有采用可控 store 模式。
- 影响：
  - 多组件复用同一个 composable 时，状态会互相污染，行为依赖调用顺序。
  - 单测之间容易串状态，页面切换和热更新也会产生诡异残留。
  - 后续如果要做多标签、多视图并存或 SSR，当前结构基本不可扩展。
- 建议：
  - 页面级状态保留在 composable 内部；确实要跨组件共享的，再升级到 Pinia store。
  - 为 Direction/Day 拆成“数据 store + 页面 action + 派生 selector”，避免模块变量充当隐式 store。
  - 为共享缓存增加显式 `reset()`，并在路由离开、登出、用户切换时调用。
- 优先级：高

## 问题 4：Direction 的缓存与初始化标志不会随用户会话重置，存在跨账号残留数据风险
- 标题：Direction 模块共享缓存没有会话级清理
- 证据：
  - [src/views/direction/composables/useDirectionState.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/composables/useDirectionState.js:37)
  - [src/views/direction/composables/useDirectionFetch.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/composables/useDirectionFetch.js:155)
  - [src/App.vue](/Users/sifan1/Documents/project/日程/rhythm/src/App.vue:72)
  - [src/stores/authStore.js](/Users/sifan1/Documents/project/日程/rhythm/src/stores/authStore.js:18)
- 问题：
  - `monthlyPlansCache`、`dailyPlansCache`、`plans`、`selectedGoal`、`initialized` 都是模块级共享状态。
  - `App.vue` 登出时只清理了 `authStore`，没有清任何业务缓存。
  - `useDirectionFetch()` 还通过 `setupDone`/`initialized` 阻止重复初始化，进一步放大残留状态问题。
- 原因：
  - 当前实现把 Direction 当作单次打开的页面，而不是用户会话中的业务模块。
  - 缺少“会话切换 -> 业务缓存失效”的基础设施。
- 影响：
  - 用户退出后，下一位用户可能短暂看到上一个用户的目标/月计划缓存。
  - 如果重新登录后 `initialized` 已经为真，页面不会按预期重新冷启动。
  - 这类问题很难稳定复现，但一旦出现就是严重的账号边界问题。
- 建议：
  - 为 Direction 增加 `resetDirectionState()`，在登出、登录用户变化、路由离开时调用。
  - 不要用模块变量 `setupDone` 管 watcher 生命周期，改成在组件实例中注册并自动销毁。
  - 如果要保留缓存，缓存 key 至少要带上 `userId`。
- 优先级：高

## 问题 5：Year 视图对 Habit 数据结构的假设与服务层返回值不一致，年度统计实际失效
- 标题：Year 页依赖 `habit.habit_logs`，但服务层从未返回该字段
- 证据：
  - [src/views/year/composables/useYearView.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/year/composables/useYearView.js:40)
  - [src/views/year/composables/useYearView.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/year/composables/useYearView.js:77)
  - [src/services/db/habits.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/db/habits.js:27)
- 问题：
  - Year 页通过 `db.habits.list()` 拉习惯。
  - 该服务实际只 `select('*')` 返回 habits 本体，没有 join `habit_logs`。
  - 但 Year 页后面却按 `habit.habit_logs || []` 聚合全年打卡。
- 原因：
  - 页面层和服务层没有共享的 DTO / schema 约束，组件凭“想象中的数据结构”编码。
  - 缺少端到端测试去验证年度视图是否真的拿到了日志。
- 影响：
  - 年度打卡热力/统计会长期显示为空或严重不准。
  - 用户会误判自己没有历史记录，产品信任感受损。
  - 维护时不容易第一时间发现，因为代码不会报错，只是默默给空数组。
- 建议：
  - 明确 Year 页的数据来源：要么单独提供 `db.habits.listWithLogs(year)`，要么并行查询 habits 与 habit_logs 后组装。
  - 为 `useYearView` 加一条集成测试，验证有日志时 `completedDays` 确实非空。
  - 为服务层返回值建立类型约束或运行时校验，避免页面“猜字段”。
- 优先级：高

## 问题 6：目标与月计划的写操作不是事务性的，失败时会产生半成品数据
- 标题：Direction 写路径存在父子数据部分成功的问题
- 证据：
  - [src/views/direction/composables/useDirectionGoals.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/composables/useDirectionGoals.js:85)
  - [src/views/direction/composables/useDirectionGoals.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/composables/useDirectionGoals.js:181)
  - [src/views/direction/composables/useDirectionGoals.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/composables/useDirectionGoals.js:235)
- 问题：
  - `handleAddGoal()` 先创建 `plan`，再并发创建多个 `monthlyPlans`。
  - `updateGoalData()` 先更新父目标，再并发创建/更新/删除月计划。
  - `handleDeleteGoal()` 先删月计划再删父目标，注释还写着“靠数据库级联删除”，但实际并不是一个原子过程。
- 原因：
  - 前端直接承担了多表编排责任，但没有事务、幂等、回滚机制。
  - 项目虽然在注释里提到 RPC 批量操作，但当前关键路径并未真正用后端原子接口承接。
- 影响：
  - 中途任一步失败，数据库会留下孤儿 `plan`、缺失月份、部分更新后的不一致状态。
  - UI 刷新后会看到“目标存在但月份不全”这类脏数据，后续修复成本高。
  - 业务越复杂，这条链路越难维护。
- 建议：
  - 把目标及其月计划的新增/更新/删除收敛成单个 RPC / Edge Function，由数据库事务保证一致性。
  - 至少为当前前端编排加入失败补偿与幂等策略，不要让 `Promise.all` 直接暴露半成功状态。
  - 把“计划树”的写模型定义成后端接口，而不是页面层脚本。
- 优先级：高

## 问题 7：习惯模块同时使用 `archived` 与 `is_archived`，数据模型已经分叉
- 标题：习惯归档字段命名不统一，读写口径不一致
- 证据：
  - [database/current-structure.md](/Users/sifan1/Documents/project/日程/rhythm/database/current-structure.md:81)
  - [src/views/habits/components/AddHabitModal.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/habits/components/AddHabitModal.vue:107)
  - [src/views/habits/components/EditHabitModal.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/habits/components/EditHabitModal.vue:170)
  - [src/views/day/composables/useDayData.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/composables/useDayData.js:99)
- 问题：
  - 新建习惯时写的是 `archived: false`。
  - 编辑归档时改的是 `is_archived`。
  - 列表和日视图过滤时又只读 `is_archived`。
- 原因：
  - 数据库模型历史演进后，没有在服务层完成统一映射。
  - 页面直接操作底层字段名，导致多个口径并存。
- 影响：
  - 新创建的习惯是否归档，取决于数据库默认值和历史字段迁移状态，行为不可预测。
  - 代码维护者很难判断应该改哪个字段，容易继续扩散脏写法。
  - 后续做查询索引、迁移脚本、统计分析都会被旧字段拖累。
- 建议：
  - 立即在服务层统一归档字段，只暴露一个业务字段，例如 `isArchived`。
  - 页面层禁止直接写 `archived` / `is_archived` 原始列名。
  - 做一次数据库清洗和迁移，确认旧字段是否还能删；在删除前至少由 adapter 做单向兼容。
- 优先级：中
