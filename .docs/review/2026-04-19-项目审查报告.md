# 项目审查报告

## 一、项目目标理解
- 项目核心目标：基于当前 `README` 和路由结构推断，这是一个把“长期目标拆解、周期习惯追踪、每日时间安排、周期复盘”串成闭环的个人效能管理系统，主线是 `Direction -> Month/Day -> Summary`。
- 目标用户：重视自我规划、习惯养成、日程执行和复盘沉淀的个人用户，尤其是愿意做长期目标管理的知识工作者。
- 核心使用场景：创建长期目标、拆分月/日计划、在日视图安排和执行具体任务、记录习惯完成情况、按日/周/月/年做总结。
- 本次审查范围：`src/views` 下的登录、日视图、月视图、年视图、习惯、所向、总结模块；`src/services`、`src/stores`、`src/composables` 的数据访问与状态管理；`README.md` 与 `database/current-structure.md`。另外执行了 `pnpm test:unit` 和 `pnpm build` 进行验证。

## 二、最影响用户体验的 3 个问题
### 1. 目标拆解闭环没有真正落到每日执行层
- 优先级：高
- 所属维度：产品 / UX / 代码
- 问题：项目叙事强调“长期目标到每日执行的闭环”，但日视图把 `task` 和 `daily_plan` 一起混排成“今天要做的事”，却没有向用户显示这些日计划属于哪个目标，也没有解释两类任务的边界。证据见 [README.md](/Users/sifan1/Documents/project/日程/rhythm/README.md)、[useDayData.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/composables/useDayData.js:78)、[Sidebar.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/components/Sidebar.vue:93)。
- 原因：数据层已经有 `daily_plans -> monthly_plans -> plans` 关系，但产品层和 UI 层没有把这条关系暴露出来；同时 `task` 与 `daily_plan` 是两套并行执行模型，没有产品定义。
- 修改建议：在日视图把 `daily_plan` 显示为“所属目标 / 所属月计划”的承诺项，普通 `task` 显示为临时事项；增加从 Day 回跳 Direction 的入口，并给用户明确的“临时任务 vs 目标任务”说明和转化动作。
- 预期收益：用户能理解今天的动作是否在推进长期目标，Direction 不再沦为独立规划板，产品主线真正闭环。

### 2. 移动端核心任务流存在明显阻塞
- 优先级：高
- 所属维度：UX / UI
- 问题：移动端编辑入口依赖 `dblclick` 与 hover 显示的齿轮按钮，新增任务又被拆成 `Quick Add` 与“完整模式”两个抽屉，用户需要先判断走哪条流。证据见 [Sidebar.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/components/Sidebar.vue:44)、[QuickAddDrawer.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/components/QuickAddDrawer.vue:21)、[day/index.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/index.vue:87)。
- 原因：桌面交互模式被直接迁移到移动端，极简视觉优先级压过了操作可发现性；录入流程为了追求“快”做了双流，却没有处理默认值风险。
- 修改建议：移动端任务卡片提供常驻“更多”按钮或单击进入详情；把新增任务合并成一个抽屉，首屏只保留标题 + 时间两个关键字段，其余折叠展开；保留快速创建时，必须显式展示默认时间/时长并允许就地修改。
- 预期收益：减少用户在手机上“能看不能改”“创建后发现时间错了”的阻塞，提升核心任务流完成率。

### 3. Direction 模块的状态、请求和缓存边界不稳定
- 优先级：高
- 所属维度：性能 / 代码 / UX
- 问题：`Direction` 在切换目标和月份时会重复请求同一份日计划，模块状态又以顶层单例和缓存存在，且登出时没有重置，会产生竞态、旧数据残留和测试失败。证据见 [useDirectionFetch.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/composables/useDirectionFetch.js:83)、[useDirectionState.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/composables/useDirectionState.js)、[authStore.js](/Users/sifan1/Documents/project/日程/rhythm/src/stores/authStore.js:18)。`pnpm test:unit` 也在 [useDirectionFetch.spec.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/composables/__tests__/useDirectionFetch.spec.js) 上失败了 2 个并发相关用例。
- 原因：同一个职责被 `selectedGoal` watcher、`selectedMonth` watcher 和初始化逻辑重复承担；状态没有按用户会话建立边界；缺少 in-flight 复用和 reset 机制。
- 修改建议：把“选默认月份 + 拉该月 daily plans”收敛为单入口；对同一 `monthlyPlanId` 复用 Promise 或加请求版本号；把 Direction 状态迁到可显式 `reset()` 的 store，并在登出/切换用户时清空缓存。
- 预期收益：减少重复请求和错误覆盖，提升目标页切换稳定性，降低后续维护成本。

## 三、详细问题清单
### 产品逻辑
#### 问题 1：年 / 月 / 日三层导航不是同一条业务线
- 优先级：高
- 问题：导航把 `/year`、`/month`、`/day` 都归在“时序”，但年视图只看习惯，月视图主要看任务，日视图又聚合任务、日计划和习惯，用户会以为这是同一对象的时间下钻，实际上不是。证据见 [Navbar.vue](/Users/sifan1/Documents/project/日程/rhythm/src/components/Navbar.vue:195)、[useYearView.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/year/composables/useYearView.js:37)、[useMonthView.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/month/composables/useMonthView.js)。
- 原因：路由按时间粒度命名，内容却按不同数据模型组织。
- 修改建议：要么统一成真正的年/月/日时间总览，要么把导航语义改成“年习惯总览 / 月任务排布 / 今日执行”，避免误导用户心智。

#### 问题 2：系统存在两套“今天要做的事”，缺少明确边界
- 优先级：高
- 问题：`task` 与 `daily_plan` 都进入日视图，但新增入口只会创建 `task`，Direction 又能生成 `daily_plan`，用户很难判断重要事项到底该建在哪。证据见 [useDayData.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/composables/useDayData.js:121)、[AddEventModal.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/components/AddEventModal.vue:18)、[useDirectionFetch.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/composables/useDirectionFetch.js:112)。
- 原因：产品层缺少“临时事项”和“目标承诺项”的角色定义。
- 修改建议：明确对象语义，并在日视图做分组或标签；增加“把普通任务升级为目标项”之类的转化动作。

#### 问题 3：缺少收件箱 / 待安排层，用户被迫立刻排时
- 优先级：中高
- 问题：无论是完整新增还是移动端 Quick Add，任务都默认带时间和时长，系统没有“先记下，再安排”的缓冲层。证据见 [AddEventModal.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/components/AddEventModal.vue:31)、[QuickAddDrawer.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/components/QuickAddDrawer.vue:34)、[router/index.js](/Users/sifan1/Documents/project/日程/rhythm/src/router/index.js:11)。
- 原因：当前产品更像排程器，而不是完整的个人任务流。
- 修改建议：增加收件箱或支持“无时间任务”，让用户先记录，再安排到具体日期或目标。

#### 问题 4：总结模块没有与执行数据形成复盘闭环
- 优先级：中
- 问题：总结页只是单独的内容表单，没有自动带出任务完成、习惯完成、目标推进事实，用户需要自己回忆。证据见 [useSummaryManager.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/summary/composables/useSummaryManager.js:41)、[summaries.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/db/summaries.js)。
- 原因：总结被设计成独立模块，而不是执行结果的复盘层。
- 修改建议：从 Day / Habits / Direction 提供“生成今日/本周总结”的入口，并预填周期事实数据。

### 用户体验（UX）
#### 问题 1：新用户首访缺少明确起步路径
- 优先级：高
- 问题：登录成功后直接进入日视图，页面没有说明四个一级模块分别干什么，也没有推荐起步动作。证据见 [login/index.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/login/index.vue:18)、[router/index.js](/Users/sifan1/Documents/project/日程/rhythm/src/router/index.js:18)、[Navbar.vue](/Users/sifan1/Documents/project/日程/rhythm/src/components/Navbar.vue:173)。
- 原因：默认假设用户理解“时序 / 所向 / 总结”的方法论。
- 修改建议：增加首次进入引导或首页说明卡，至少给“先加今天第一件事 / 先创建一个目标 / 先添加习惯”三个明确入口。

#### 问题 2：移动端任务编辑入口不可发现
- 优先级：高
- 问题：移动端任务卡片把编辑动作隐藏在 hover 按钮和 `dblclick` 上，这在触屏场景里几乎不可用。证据见 [Sidebar.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/components/Sidebar.vue:45)。
- 原因：桌面交互心智被直接迁移到手机。
- 修改建议：使用常驻更多按钮、长按菜单或单击进入详情，显式暴露编辑能力。

#### 问题 3：提交、删除、失败后的反馈不完整
- 优先级：高
- 问题：任务、总结、习惯等多个流程失败时主要是 `console.error`，删除确认和成功反馈也不统一，用户很难判断操作是否生效。证据见 [useAddEventForm.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/composables/useAddEventForm.js)、[useSummaryManager.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/summary/composables/useSummaryManager.js:70)、[safeDb.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/safeDb.js:72)。
- 原因：工程层考虑了异常捕获，但交互层没有完整建模“提交中 / 成功 / 失败 / 危险确认”。
- 修改建议：统一所有写操作的 loading、成功 toast、失败提示和删除确认，不要只在控制台输出错误。

#### 问题 4：空状态文案与真实布局不一致
- 优先级：中
- 问题：日视图空状态是英文，Summary 页在移动端仍提示“从左侧列表选择”，和实际纵向布局不一致。证据见 [Sidebar.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/components/Sidebar.vue:125)、[summary/index.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/summary/index.vue:39)。
- 原因：空状态只是占位，没有按实际任务和端形态设计。
- 修改建议：统一中文、面向下一步行动的空状态文案，并根据端形态调整方向描述。

#### 问题 5：通知权限请求时机过早
- 优先级：中
- 问题：用户进入应用和日视图时就会请求通知权限，往往发生在用户还没理解产品价值之前。证据见 [App.vue](/Users/sifan1/Documents/project/日程/rhythm/src/App.vue:67)、[day/index.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/index.vue:183)。
- 原因：把能力初始化放在页面加载阶段，而不是用户显式表达提醒需求之后。
- 修改建议：延迟到用户创建带时间任务、主动开启提醒、或进入设置页时再请求权限。

### UI 设计
#### 问题 1：日期与数字标题抢走了主内容注意力
- 优先级：高
- 问题：日视图侧栏把日期做成超大字号，月视图日期格也非常突出，但任务、状态和完成度没有形成稳定的二级视觉层。证据见 [Sidebar.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/components/Sidebar.vue:22)。
- 原因：海报化排版被直接用于高密度业务页面。
- 修改建议：下调日期字号，提升任务量、状态、当前日高亮和行动按钮的层级。

#### 问题 2：大量交互入口被隐藏在 hover 态
- 优先级：高
- 问题：桌面导航、任务编辑、习惯设置、目标设置等入口大量默认不可见。证据见 [Navbar.vue](/Users/sifan1/Documents/project/日程/rhythm/src/components/Navbar.vue:8)、[Sidebar.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/components/Sidebar.vue:79)、[HabitSidebar.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/habits/components/HabitSidebar.vue:35)、[DirectionSidebar.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/components/DirectionSidebar.vue:175)。

#### 问题 3：全局灰阶过淡，边界和状态不够清晰
- 优先级：高
- 问题：边框、辅助文字、弱按钮和卡片背景集中在很接近的灰阶，许多元素接近“禁用态”。证据见 [tw-theme.css](/Users/sifan1/Documents/project/日程/rhythm/src/assets/tw-theme.css)、[login/index.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/login/index.vue:23)、[DayCell.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/month/components/DayCell.vue)。
- 原因：全局 token 已经偏淡，局部又叠加透明度。
- 修改建议：拉开 `border`、`muted`、`secondary` 的梯度，减少 `/30`、`/40` 透明度叠加。

#### 问题 4：模块之间视觉语法不统一
- 优先级：中
- 问题：Day 页是杂志化强风格，Habits / Direction 更像 SaaS 工作台，Summary 是标准中台表单，Login 又是最基础登录框，整体有拼接感。证据见 [Sidebar.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/components/Sidebar.vue:22)、[summary/index.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/summary/index.vue:49)、[login/index.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/login/index.vue:18)。
- 原因：主题 token 统一了颜色，但没有统一版式与组件语法。
- 修改建议：先确定一个主视觉方向，再统一标题、侧栏项、主按钮、表单和空状态样式。

### 性能
#### 问题 1：日视图秒级定时刷新且未清理
- 优先级：高
- 问题：日视图 `setInterval(updateCurrentHour, 1000)` 每秒更新一次当前时间，但没有在卸载时清理。证据见 [useDayNavigation.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/composables/useDayNavigation.js:127)。
- 原因：时间轴刷新粒度过细，且缺少生命周期收口。
- 修改建议：降低到 30 秒或 60 秒刷新，保存 interval id 并在 `onUnmounted` 清理，只让时间标记独立更新。

#### 问题 2：日视图写操作后统一全量重取四份数据
- 优先级：高
- 问题：勾选任务、习惯或日计划后，都会重新拉 `tasks + dailyPlans + habits + habitLogs`，再整份重算 `dailySchedule`。证据见 [useDayData.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/composables/useDayData.js:68)、[useDayData.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/composables/useDayData.js:271)。
- 原因：更新策略是“写完整页刷新”，没有局部 patch。
- 修改建议：按数据集拆分刷新，给高频交互加 optimistic update，`habits.listLite()` 走会话缓存。

#### 问题 3：Direction 切换目标时会重复请求并产生竞态
- 优先级：高
- 问题：切换目标时 `selectedGoal` watcher 和 `selectedMonth` watcher 都可能触发 `loadDailyPlans(..., { force: true })`。证据见 [useDirectionFetch.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/composables/useDirectionFetch.js:162)、[useDirectionFetch.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/composables/useDirectionFetch.js:180)。
- 原因：初始化月选择和月数据预加载的职责重复。
- 修改建议：收敛单入口，复用 in-flight Promise，并加版本控制或 abort 防护。

#### 问题 4：习惯页打卡后触发双重刷新
- 优先级：高
- 问题：打卡后先 `fetchHabits()`，随后又因为 `selectedHabit` 变化重新 `fetchLogsForHabit()`，一次操作会触发两轮请求和整页重算。证据见 [useHabitData.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/habits/composables/useHabitData.js:129)、[useHabitLogs.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/habits/composables/useHabitLogs.js)。
- 原因：列表刷新和详情刷新没有解耦，且 watch 依赖对象身份变化。
- 修改建议：局部 patch 当前习惯数据，watch 改成只监听 `selectedHabit?.id`。

#### 问题 5：习惯日志没有分页，历史越久越慢
- 优先级：中
- 问题：单个习惯的全部日志一次性查出、前端全量排序、最后完整渲染。证据见 [habits.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/db/habits.js:42)、[useHabitData.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/habits/composables/useHabitData.js:166)、[HabitLogs.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/habits/components/HabitLogs.vue)。
- 原因：服务层只有全量查询接口，展示层也按全量列表处理。
- 修改建议：默认只拉最近 30-50 条，提供“加载更多”或虚拟列表。

### 代码质量 / 架构
#### 问题 1：`safeDb` 的错误契约不一致，失败会被伪装成“空结果”
- 优先级：高
- 问题：标准 CRUD 失败时返回 `null`，非标准方法又直接抛异常，调用方难以区分“真空数据”和“请求失败”。证据见 [safeDb.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/safeDb.js:72)。
- 原因：只有 `list/create/update/delete` 被包装，其它方法透传原始行为。
- 修改建议：统一改为 `throw` 或统一返回 `{ data, error }`，toast 放到 action 层。

#### 问题 2：多个 composable 把页面数据做成模块单例，实例边界丢失
- 优先级：高
- 问题：Day、日报、Direction 都有大量模块级 `ref/reactive` 状态，生命周期和会话边界不清晰。证据见 [useDayData.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/composables/useDayData.js:24)、[useDailyReport.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/composables/useDailyReport.js)、[useDirectionState.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/composables/useDirectionState.js)。
- 原因：为了“减少重复请求”直接把页面状态提到了模块顶层。
- 修改建议：页面内状态收回 composable，真正共享的状态迁到 Pinia 并提供 `reset()`。

#### 问题 3：Direction 缓存不会随用户会话重置
- 优先级：高
- 问题：登出时只清掉了 `authStore`，Direction 的缓存、初始化标记和选择态仍保留。证据见 [useDirectionFetch.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/composables/useDirectionFetch.js:209)、[App.vue](/Users/sifan1/Documents/project/日程/rhythm/src/App.vue:80)、[authStore.js](/Users/sifan1/Documents/project/日程/rhythm/src/stores/authStore.js:18)。
- 原因：缓存是按页面打开生命周期设计，而不是按用户会话设计。
- 修改建议：增加 `resetDirectionState()`，在登出和用户切换时清空缓存，缓存 key 至少带 `userId`。

#### 问题 4：查询层没有显式按用户隔离
- 优先级：高
- 问题：核心表服务层几乎都没有显式 `user_id` 过滤，安全性完全依赖 RLS。证据见 [tasks.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/db/tasks.js:16)、[plans.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/db/plans.js:17)、[monthlyPlans.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/db/monthlyPlans.js:17)、[dailyPlans.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/db/dailyPlans.js:18)、[habits.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/db/habits.js:23)。
- 原因：`createBase()` 是通用 CRUD 工厂，没有用户上下文。
- 修改建议：在 repository / service 层统一注入当前用户过滤，形成“代码层过滤 + RLS”双保险。

#### 问题 5：Year 页对 Habit 返回结构的假设与服务层不一致
- 优先级：高
- 问题：Year 页依赖 `habit.habit_logs` 计算年度打卡，但 `db.habits.list()` 只返回 `habits.*`。证据见 [useYearView.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/year/composables/useYearView.js:37)、[habits.js](/Users/sifan1/Documents/project/日程/rhythm/src/services/db/habits.js:23)。
- 原因：页面层在猜返回结构，没有 DTO / schema 约束。
- 修改建议：新增 `listWithLogs(year)` 或并行查询 `habits + habit_logs` 后再组装，并补集成测试。

#### 问题 6：Direction 父子写入不是事务性的
- 优先级：高
- 问题：创建、更新、删除目标时，`plan` 和 `monthlyPlans` 分多次请求执行，中途失败会留下半成品。证据见 [useDirectionGoals.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/direction/composables/useDirectionGoals.js:72)。
- 原因：前端直接承担多表编排，没有事务、回滚和幂等控制。
- 修改建议：把这类写操作收口为单个 RPC / Edge Function，由数据库事务保证一致性。

#### 问题 7：习惯归档字段已经分叉
- 优先级：中
- 问题：数据库同时存在 `archived` 和 `is_archived`，前端不同地方写不同字段，读取又只看 `is_archived`。证据见 [database/current-structure.md](/Users/sifan1/Documents/project/日程/rhythm/database/current-structure.md:66)、[AddHabitModal.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/habits/components/AddHabitModal.vue)、[EditHabitModal.vue](/Users/sifan1/Documents/project/日程/rhythm/src/views/habits/components/EditHabitModal.vue)、[useDayData.js](/Users/sifan1/Documents/project/日程/rhythm/src/views/day/composables/useDayData.js:99)。
- 原因：历史字段没有被服务层收口。
- 修改建议：服务层统一抽象成 `isArchived`，页面禁止直接写底层列名，并安排数据清洗。

#### 问题 8：测试已经暴露出 Direction 抓取行为与设计预期不一致
- 优先级：高
- 问题：`pnpm test:unit` 中 `useDirectionFetch.spec.js` 有 2 个并发相关用例失败，分别是“每个 plan 都应启动 monthly 请求”和“每个 monthly plan 都应启动 daily 请求”。这说明当前数据抓取实现与团队预期已经分裂。
- 原因：Direction 抓取逻辑经历过优化或改写，但测试和实现没有统一。
- 修改建议：先明确当前期望行为，再修实现或修测试，不要让失败测试长期挂着。

## 四、跨维度关键问题
### 目标系统与执行系统割裂
- 影响维度：产品 / UX / 代码
- 说明：Direction 定义了长期目标和月/日计划，但 Day 视图没有把这些执行项与目标关系讲清楚，`task` 与 `daily_plan` 又并行存在，导致用户心智混乱，产品主线断裂。
- 建议优先处理方式：先统一对象模型，再在 Day 视图显式展示来源和分组，最后补上从 Day 到 Direction 的双向跳转。

### 交互入口隐藏过度
- 影响维度：UX / UI / 新手可用性
- 说明：导航、编辑、设置、更多操作大量依赖 hover 和弱提示，桌面可发现性已经偏低，移动端更直接失效。
- 建议优先处理方式：梳理所有高频动作，先把“新增、编辑、切换、更多”四类入口做成默认可见的弱显控件。

### 状态和缓存边界不清
- 影响维度：性能 / 代码质量 / 数据正确性
- 说明：模块级单例、重复 watcher、safeDb 错误契约不一致、缓存不按用户隔离，叠加后容易出现旧数据残留、重复请求、失败伪装成空数据的问题。
- 建议优先处理方式：先定义统一的错误协议和 reset 机制，再重构 Direction / Day 的状态拥有者与数据拉取入口。

### 反馈设计缺失
- 影响维度：UX / 信任感 / 可维护性
- 说明：多个写操作缺少统一 loading、成功提示、失败提示和危险确认；工程层只打日志，交互层没有把反馈做完整。
- 建议优先处理方式：建立统一的 action feedback 规范，并在任务、习惯、总结三个高频写入口先落地。

## 五、修复优先级建议
### 第一批：必须马上修
- 打通 `Direction -> Day` 的目标可见性，明确 `task` 与 `daily_plan` 的产品边界。
- 修复移动端任务编辑入口与新增任务双流问题，确保手机上可以自然地增删改。
- 重构 `useDirectionFetch` 的请求入口、缓存重置和竞态控制，顺手修复当前失败单测。
- 给所有高频写操作补齐 loading、成功/失败反馈和删除确认。
- 修复 Year 页对 `habit_logs` 的错误假设，避免年度总览长期显示假数据。

### 第二批：建议近期修
- 收拢 `safeDb` 错误契约，避免 `null` 和异常混用。
- 把页面级模块单例状态迁回 composable / store，并增加 `reset()`。
- 降低日视图时间线刷新频率并清理定时器。
- 给习惯日志加分页或最近记录窗口，避免历史数据越多越慢。
- 重写年 / 月 / 日的信息架构文案，让用户理解每层到底在看什么。

### 第三批：可以后续打磨
- 统一视觉语法，减少不同模块之间的拼接感。
- 优化灰阶 token 和层级对比度，提升可读性和可点击感。
- 增加首访 onboarding、空状态引导和通知权限延迟请求。
- 增加收件箱 / 待安排层，补全“收集 -> 规划 -> 执行 -> 复盘”工作流。

## 六、总结结论
- 当前项目最大的问题是什么：问题主要不在产品方向，而在执行层。方向本身是对的，但“目标、执行、复盘”没有被做成一条连贯工作流；同时 Direction 与 Day 的状态和请求边界不稳，已经开始影响正确性、性能和可维护性。
- 当前项目最值得保留的优点是什么：模块划分本身很有潜力，`Direction / Habits / Day / Summary` 这四块拼图是成立的；代码里也能看到对异构数据聚合、移动端抽屉、日程视图和目标拆解的明确设计意识，说明产品不是没有方法论，而是落地还不够收口。
- 如果只能改 1~3 个点，最应该先改什么：第一，统一 `task` 与 `daily_plan` 的模型，并把目标关系带到日视图；第二，修掉移动端编辑入口和新增任务双流；第三，重构 Direction 的抓取、缓存和会话重置逻辑，让目标页先稳定下来。
